>[!question]- Приватный vs публичный ключ — в чём разница
>SSH использует **асимметричную криптографию**: ключи работают в паре.
>
>| | Приватный (`nikitinssh`) | Публичный (`nikitinssh.pub`) |
>| где хранится | только у владельца, защищён | можно свободно раздавать (сервер, GitHub) |
>| доступ | строго секретно (часто + passphrase) | открыто |
>| формат файла | без `.pub` (`id_ed25519`, `nikitinssh`) | с `.pub` |
>| восстановление | потерян → доступ пропал | можно вывести заново из приватного |
>| функция (в SSH-auth) | **ПОДПИСЫВАЕТ** | **ПРОВЕРЯЕТ подпись** |
>
>⚠️ Уточнение: для SSH-аутентификации это именно **подпись/проверка** (challenge-response), НЕ «шифрование/расшифровка данных». Текстбучная формула «приватный расшифровывает» относится к RSA-шифрованию; современная SSH-аутентификация (особенно Ed25519) — только подпись.
>Приватный из публичного вывести НЕЛЬЗЯ (наоборот — можно).

>[!question]- Как выглядят ключи
>```
># Приватный (nikitinssh):
>-----BEGIN OPENSSH PRIVATE KEY-----
>b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAA...
>-----END OPENSSH PRIVATE KEY-----
>
># Публичный (nikitinssh.pub):
>ssh-ed25519 AAAAC3NzaC1lZDI1NTE5... user@host
>```
>Первое слово публичного (`ssh-ed25519` / `ssh-rsa`) — тип ключа.

>[!question]- Что такое ssh-agent
>Программа, держащая **расшифрованные приватные ключи в памяти** и выполняющая подписание по запросу. Смысл: ввести passphrase один раз, дальше агент сам подписывает — не вводить пароль на каждый `git push`. Ключ при этом не покидает агент.
>```bash
>ssh-add ~/.ssh/nikitinssh   # добавить ключ в агент
>ssh-add -l                  # список ключей в агенте (пусто/ошибка → агент не используется)
>``` ^4d50b7

>[!question]- Как происходит подключение к серверу (handshake)
>```
>1. TCP на порт 22
>2. Transport: согласование алгоритмов + Diffie-Hellman → общий СИММЕТРИЧНЫЙ
>   сессионный ключ; сервер доказывает свою личность host-key'ом (known_hosts).
>   → дальше канал ЗАШИФРОВАН
>3. Аутентификация (publickey):
>   - клиент: «вхожу по ключу с отпечатком X»
>   - сервер: ключ X есть среди разрешённых? (authorized_keys / настройки аккаунта)
>   - клиент: ПОДПИСЫВАЕТ приватным ключом данные сессии (session id + запрос)
>   - сервер: проверяет подпись ПУБЛИЧНЫМ ключом X
>4. подпись верна → доступ
>```
>Сервер не «получает ключ и сравнивает» — он **проверяет подпись** публичным ключом (как `Signature.verify(publicKey)`, см. [[JWT]] про RS256/ES256).

>[!question]- При git push приватный ключ отправляется на сервер?
>**НЕТ, никогда.** Локально приватным ключом считается **подпись, привязанная к текущей сессии**, и на сервер уходит ОНА, а не ключ.
>- подписывает не `git`, а `ssh`, которого git вызывает под капотом;
>- passphrase тоже не уходит — она лишь локально расшифровывает ключ.
>
>Replay невозможен: подпись над данными конкретной сессии (уникальный session id из Diffie-Hellman) → в другой сессии не подойдёт.

>[!question]- Кто вычисляет подпись — ssh-agent или ssh-клиент?
>Зависит (агент опционален):
>- **ключ в ssh-agent** → подпись считает АГЕНТ: ssh-клиент шлёт ему по сокету данные, агент подписывает ключом из памяти, возвращает подпись. Ключ не уходит даже в процесс ssh-клиента.
>- **агента нет / ключ не добавлен** → подпись считает САМ ssh-клиент: читает файл `~/.ssh/nikitinssh`, при необходимости расшифровывает passphrase'ой, подписывает.
>Результат одинаковый (на сервер — подпись), разница лишь в том, какой локальный процесс держит ключ.

>[!question]- Какие git-команды вообще используют SSH
>- **Локальные (без SSH):** `commit`, `add`, `log`, `status`, `branch`, `diff` — сети не касаются.
>- **Сетевые (через SSH):** `push`, `pull`, `fetch`, `clone`, `ls-remote` — открывают SSH-сессию.
>Каждая сетевая команда = новая сессия + новый handshake + новая подпись (ключ всё равно не передаётся). Оптимизации: ssh-agent (не вводить passphrase), ControlMaster/multiplexing (переиспользовать соединение).

>[!question]- Что реально «летит по проводу»
>| Сущность | Где живёт | Передаётся серверу? |
>| Приватный ключ | только у клиента | ❌ никогда |
>| Passphrase | только у клиента | ❌ никогда |
>| Публичный ключ | у клиента + на сервере | заранее, один раз при настройке |
>| Подпись над данными сессии | вычисляется на лету | ✅ каждую сессию (новая) |

>[!question]- Правило безопасности
>🔒 Приватный ключ = как пароль: никому не передавать, не коммитить в git, права `600`, хранить в `~/.ssh/`.
>🔓 Публичный — можно раздавать серверам/Git-хостам.
