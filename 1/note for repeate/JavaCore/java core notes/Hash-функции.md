# Hash-функции (криптографические)

Пример: `com.example.testlinux.security.HashExample`

> [!info] Не путать с `Object.hashCode()`
> `hashCode()` — это хеш **для коллекций** (HashMap/HashSet), быстрый, 32 бита, коллизии норма.
> Здесь речь про **криптографические** хеши (SHA-1, SHA-256) — медленные по дизайну, односторонние, для безопасности (пароли, цифровые подписи, контроль целостности).
> Про `hashCode()` см. [[JAVA-HASHMAP]].

---

> [!question]- Что такое криптографическая хеш-функция?
> Функция, которая превращает данные **любой** длины в строку **фиксированной** длины (дайджест). Свойства:
> - **детерминированность** — один и тот же вход всегда даёт один и тот же выход;
> - **односторонность** — по хешу нельзя восстановить исходные данные;
> - **лавинный эффект** — изменение одного бита входа меняет ~половину битов выхода;
> - **стойкость к коллизиям** — практически невозможно найти два разных входа с одинаковым хешем.
> Длина выхода фиксирована: SHA-1 → 160 бит (40 hex-символов), SHA-256 → 256 бит (64 hex-символа).

> [!question]- Как посчитать хеш строки в Java?
> Через `java.security.MessageDigest`:
> ```java
> MessageDigest digest = MessageDigest.getInstance("SHA-256");
> byte[] hash = digest.digest(input.getBytes(StandardCharsets.UTF_8));
> ```
> - `getInstance("SHA-256")` — выбираем алгоритм по имени ("SHA-1", "SHA-256", "MD5", ...). Кидает `NoSuchAlgorithmException`, если алгоритма нет.
> - `digest(byte[])` — считает хеш и возвращает массив байт.
> - **Важно явно задать кодировку** `StandardCharsets.UTF_8`: иначе `getBytes()` берёт платформо-зависимую кодировку, и хеш одной и той же строки может отличаться на разных машинах.

> [!question]- Зачем переводить хеш в Hex-строку и как это сделать?
> `digest()` возвращает «сырые» байты — их неудобно хранить и сравнивать (непечатаемые символы). Поэтому каждый байт переводят в две шестнадцатеричные цифры:
> ```java
> StringBuilder hexString = new StringBuilder();
> for (byte b : encodedHash) {
>     String hex = Integer.toHexString(0xff & b);
>     if (hex.length() == 1) hexString.append('0');  // ведущий ноль
>     hexString.append(hex);
> }
> ```
> Итог — строка вида `290c8ce73bc0...` фиксированной длины (64 символа для SHA-256).

> [!question]- Зачем нужна маска `0xff & b` при переводе байта в hex?
> В Java `byte` **знаковый** (диапазон −128..127). При расширении до `int` отрицательный байт получает знаковое расширение — старшие биты заполняются единицами:
> ```
> byte b = (byte) 0x8F;   // = -113
> Integer.toHexString(b); // "ffffff8f"  ← мусор!
> ```
> `0xff & b` обнуляет старшие 24 бита и оставляет только младший байт (0..255):
> ```
> Integer.toHexString(0xff & b); // "8f"  ← правильно
> ```

> [!question]- Зачем проверка `if (hex.length() == 1) hexString.append('0')`?
> `Integer.toHexString` **не добавляет ведущий ноль**: байт `0x05` даёт `"5"`, а не `"05"`. Без выравнивания до 2 символов hex-строка «съедет» и хеши будут сравниваться неверно. Поэтому к однозначным значениям дописываем ведущий `0`.

> [!question]- Что такое соль (salt) и зачем она в примере?
> Соль — случайная строка, которую добавляют к паролю **перед** хешированием:
> ```java
> String input = salt + password;            // "qX7z!" + "bgn"
> byte[] hash = digest.digest(input.getBytes(UTF_8));
> ```
> Зачем:
> - **защита от радужных таблиц** — заранее посчитанные таблицы «хеш → пароль» становятся бесполезны, т.к. соль уникальна;
> - **разные хеши у одинаковых паролей** — два пользователя с паролем `123` получат разные хеши, если соль у каждого своя.
> На практике соль генерируют случайно (`SecureRandom`), хранят рядом с хешем, и она у **каждого** пользователя своя.

> [!question]- Почему SHA-1 и SHA-256 нельзя использовать для хранения паролей?
> Они спроектированы **быстрыми** — это плюс для контроля целостности, но минус для паролей: атакующий перебирает миллиарды вариантов в секунду на GPU. Для паролей нужны **намеренно медленные** и «дорогие» по памяти алгоритмы:
> - **bcrypt**, **scrypt**, **Argon2** (рекомендуется), **PBKDF2**.
> Они встроенно делают соль + тысячи итераций. В Spring Security — `BCryptPasswordEncoder` / `Argon2PasswordEncoder`.
> SHA-1 вдобавок **взломан** (найдены коллизии, 2017) — для безопасности использовать нельзя вообще.

> [!question]- Чем отличаются SHA-1 и SHA-256?
> | | SHA-1 | SHA-256 |
> |---|---|---|
> | Длина хеша | 160 бит (40 hex) | 256 бит (64 hex) |
> | Стойкость | **взломан** (коллизии) | стойкий |
> | Применение | только legacy/контроль целостности | подписи, сертификаты, блокчейн |
>
> Оба — детерминированные и односторонние, но SHA-256 из семейства SHA-2 и на сегодня безопасен.

> [!question]- Почему `MessageDigest` нельзя переиспользовать наивно между потоками?
> `MessageDigest` **хранит состояние** (накопленные данные) и **не потокобезопасен**. После `digest()` объект автоматически сбрасывается (`reset()`) и готов к новому хешированию, но один и тот же экземпляр нельзя дёргать из нескольких потоков одновременно. В многопоточном коде — создавай новый `getInstance(...)` на каждый вызов или используй `ThreadLocal`.

## Хеширование паролей (Spring Security)

Пример: `com.example.testlinux.security.PasswordEncoderExample`

> [!tip] Главная идея
> SHA-* — **быстрые** (плюс для контроля целостности, минус для паролей). Алгоритмы ниже **намеренно медленные / дорогие по памяти** — это и есть защита от перебора на GPU. Все они реализуют интерфейс `PasswordEncoder` (`encode` / `matches`).

> [!question]- Как устроен интерфейс `PasswordEncoder`?
> Два метода:
> - `String encode(CharSequence raw)` — хеширует; **соль генерируется внутри** и встраивается в строку результата;
> - `boolean matches(CharSequence raw, String encoded)` — проверка; соль и параметры берутся **из самой строки** `encoded`.
> Следствие: один и тот же пароль каждый раз даёт **разный** хеш (соль случайная), поэтому сравнивать хеши «в лоб» через `equals` нельзя — только через `matches()`.

> [!question]- BCrypt — что это и как читать его хеш?
> Отраслевой стандарт по умолчанию, основан на шифре Blowfish.
> ```java
> BCryptPasswordEncoder enc = new BCryptPasswordEncoder(); // strength=10
> String hash = enc.encode(password);
> boolean ok = enc.matches(password, hash);
> ```
> Формат: `$2a$10$<22 символа соли><31 символ хеша>`
> - `$2a$` — версия, `10` — **strength** (cost / log-rounds): 2^10 = 1024 итерации; каждое +1 **удваивает** время.
> Ограничение: учитывает только **первые 72 байта** пароля.

> [!question]- Argon2 — почему рекомендуется для нового кода?
> Победитель Password Hashing Competition (2015). **Memory-hard** — требует много памяти, поэтому стоек к атакам на GPU/ASIC (там память дорогая).
> ```java
> Argon2PasswordEncoder enc = Argon2PasswordEncoder.defaultsForSpringSecurity_v5_8();
> ```
> Формат: `$argon2id$v=19$m=16384,t=2,p=1$<соль>$<хеш>`
> - `m` — память (KB), `t` — число итераций, `p` — параллелизм. Вариант **argon2id** — рекомендуемый.
> Требует **BouncyCastle** (`bcprov`) на classpath.

> [!question]- SCrypt — чем отличается от Argon2?
> Тоже **memory-hard**, предшественник Argon2 (используется в т.ч. в криптовалютах).
> ```java
> SCryptPasswordEncoder enc = SCryptPasswordEncoder.defaultsForSpringSecurity_v5_8();
> ```
> Формат: `$<params>$<соль>$<хеш>` (base64). Параметры: `N` (cpuCost), `r` (memoryCost), `p` (parallelization).
> Тоже требует **BouncyCastle**.

> [!question]- PBKDF2 — когда применять?
> Старый, но всё ещё одобренный (NIST/FIPS) алгоритм. **НЕ memory-hard** — уязвимее к GPU, чем Argon2/SCrypt, но прост и встроен в JDK (не нужен BouncyCastle).
> ```java
> Pbkdf2PasswordEncoder enc = Pbkdf2PasswordEncoder.defaultsForSpringSecurity_v5_8(); // 310000 итераций, SHA-256
> ```
> Берёт HMAC-SHA256 и прогоняет N итераций. Выбирать, когда нужна **FIPS-совместимость** или нельзя тянуть зависимости.

> [!question]- Что такое `DelegatingPasswordEncoder` и зачем префиксы `{bcrypt}`?
> Это то, что Spring Boot ставит бином `PasswordEncoder` **по умолчанию** (`PasswordEncoderFactories.createDelegatingPasswordEncoder()`).
> Хеш получает префикс с именем алгоритма:
> ```
> {bcrypt}$2a$10$...
> {argon2}$argon2id$...
> ```
> Зачем:
> - **миграция алгоритмов без сброса паролей** — новые пароли пишутся текущим алгоритмом, старые проверяются по своему префиксу;
> - `matches()` сам выбирает нужный энкодер по префиксу (проверит даже хеш, сделанный другим алгоритмом).

> [!question]- Сравнение алгоритмов хеширования паролей
> | Алгоритм | Memory-hard | Зависимости | Когда |
> |---|---|---|---|
> | **BCrypt** | нет | нет (в spring-security-crypto) | дефолт, хороший выбор |
> | **Argon2** (argon2id) | **да** | BouncyCastle | **рекомендуется** для нового кода |
> | **SCrypt** | **да** | BouncyCastle | альтернатива Argon2 |
> | **PBKDF2** | нет | нет (JDK) | FIPS / минимум зависимостей |
> | **SHA-256** | — | нет (JDK) | ⛔ **не для паролей** (быстрый) |
>
> На практике: используй `DelegatingPasswordEncoder`, основной алгоритм — BCrypt или Argon2id.
