# Паттерн проектирования Factory
1. [ ] Книга делится на три категории шаблонов: **порождающие**, **структурные** и **поведенческие**. ## Design Patterns: Elements of Reusable Object-Oriented Software

>[!question]- Что такое Factory?
>**Factory (Фабричный метод)** — порождающий паттерн, который определяет интерфейс для создания объектов, но позволяет подклассам решать, какой класс инстанциировать.
>
>**Зачем:** скрыть логику создания объектов, убрать `new ConcreteClass()` из бизнес-кода, легко добавлять новые типы без изменения существующего кода.
>
>**Пример на Java:**
>```java
>// Продукт
>interface Animal {
>    void speak();
>}
>
>class Dog implements Animal {
>    public void speak() { System.out.println("Woof!"); }
>}
>
>class Cat implements Animal {
>    public void speak() { System.out.println("Meow!"); }
>}
>
>// Фабрика
>class AnimalFactory {
>    public static Animal create(String type) {
>        return switch (type) {
>            case "dog" -> new Dog();
>            case "cat" -> new Cat();
>            default -> throw new IllegalArgumentException("Unknown: " + type);
>        };
>    }
>}
>
>// Использование
>Animal a = AnimalFactory.create("dog");
>a.speak(); // Woof!
>```
>
>**Виды:**
>- **Simple Factory** — статический метод создаёт объекты (пример выше)
>- **Factory Method** — абстрактный метод в базовом классе, подкласс переопределяет
>- **Abstract Factory** — фабрика фабрик, создаёт семейства связанных объектов

>[!question]- Simple Factory vs Factory Method — отличия
>**Simple Factory** — не паттерн GoF, просто идиома. Один класс со статическим методом и `switch/if` внутри.
>**Factory Method** — официальный паттерн GoF. Абстрактный метод в базовом классе, подкласс решает что создавать.
>
>| | Simple Factory | Factory Method |
>|---|---|---|
>| Паттерн GoF | нет | да |
>| Как добавить новый тип | изменить фабрику | создать новый подкласс |
>| Open/Closed принцип | нарушает | соблюдает |
>| Логика создания | в одном методе | размазана по подклассам |
>| Когда использовать | мало типов, редко меняется | типов много, часто расширяется |
>
>**Simple Factory:**
>```java
>class TransportFactory {
>    public static Transport create(String type) {
>        return switch (type) {
>            case "car"  -> new Car();
>            case "bike" -> new Bike();
>            // добавить Truck? — лезем СЮДА и меняем этот класс
>            default -> throw new IllegalArgumentException(type);
>        };
>    }
>}
>```
>
>**Factory Method:**
>```java
>abstract class Logistics {
>    abstract Transport createTransport(); // фабричный метод
>
>    void deliver() {
>        Transport t = createTransport(); // не знает, что именно
>        t.move();
>    }
>}
>
>class RoadLogistics extends Logistics {
>    Transport createTransport() { return new Truck(); }
>}
>
>class SeaLogistics extends Logistics {
>    Transport createTransport() { return new Ship(); }
>}
>// добавить AirLogistics? — новый класс, старый код не трогаем
>```
>
>**Ключевое отличие одной фразой:**
>Simple Factory — это `if/switch` в одном месте. Factory Method — это полиморфизм вместо `if/switch`.

>[!question]- Три паттерна связанных с фабрикой
>Все три про создание объектов, но на разном уровне сложности:
>
>**1. Simple Factory** — не GoF, просто идиома
>Один класс, один статический метод, `if/switch` внутри. Создаёт один тип объекта.
>
>**2. Factory Method** — GoF, порождающий
>Абстрактный метод в базовом классе. Подкласс переопределяет и решает что создавать. Обычно один фабричный метод.
>
>**3. Abstract Factory** — GoF, порождающий
>Несколько фабричных методов. Создают **семейство связанных объектов** (кнопка + диалог + иконка — всё в одном стиле).
>
>**Эволюция:**
>```
>Simple Factory → Factory Method → Abstract Factory
>  (if/switch)     (1 метод,         (много методов,
>                   полиморфизм)      семейство объектов)
>```
>Чем правее — тем больше гибкости, но и сложности.

>[!question]- Что такое фабричный метод? Нужен ли абстрактный класс?
>**Фабричный метод** — это тот метод, который **создаёт и возвращает объект**.
>Абстрактных методов в классе может быть сколько угодно, фабричным называется именно тот, что отвечает за создание.
>
>**Абстрактный класс — НЕ обязателен.** Важно различать две роли:
>- **Продукт** — то, что создаётся (`interface Transport`) — почти всегда интерфейс
>- **Создатель** — тот, у кого есть фабричный метод — может быть и абстрактным классом, и интерфейсом
>
>---
>
>**Вариант 1: Создатель — абстрактный класс** (есть общая логика)
>
>Используй когда подклассы должны **разделять одинаковое поведение** — оно живёт в базовом классе, переопределять его не нужно.
>
>```java
>// Продукт
>interface Transport {
>    void move();
>}
>class Truck implements Transport {
>    public void move() { System.out.println("Truck driving on road"); }
>}
>class Ship implements Transport {
>    public void move() { System.out.println("Ship sailing on sea"); }
>}
>
>// Создатель — абстрактный класс с ОБЩЕЙ логикой
>abstract class Logistics {
>    abstract Transport createTransport(); // фабричный метод — подкласс обязан реализовать
>
>    // общая логика: оба подкласса используют этот метод БЕЗ переопределения
>    void deliver() {
>        System.out.println("Preparing shipment...");
>        Transport t = createTransport(); // вызывает фабричный метод
>        t.move();
>        System.out.println("Delivered!");
>    }
>}
>
>// Подклассы — реализуют только фабричный метод, deliver() берут из базового класса
>class RoadLogistics extends Logistics {
>    Transport createTransport() { return new Truck(); } // своя реализация
>}
>
>class SeaLogistics extends Logistics {
>    Transport createTransport() { return new Ship(); } // своя реализация
>}
>
>// Использование
>Logistics l = new RoadLogistics();
>l.deliver();
>// Preparing shipment...
>// Truck driving on road
>// Delivered!
>// ^^^ строки "Preparing" и "Delivered" — общая логика из базового класса
>// ^^^ "Truck driving" — результат фабричного метода подкласса
>```
>
>---
>
>**Вариант 2: Создатель — интерфейс** (общей логики нет, каждый сам по себе)
>
>Используй когда подклассы **полностью независимы** и не должны ничего разделять.
>
>```java
>// Продукт
>interface Transport {
>    void move();
>}
>class Truck implements Transport {
>    public void move() { System.out.println("Truck driving"); }
>}
>class Drone implements Transport {
>    public void move() { System.out.println("Drone flying"); }
>}
>
>// Создатель — интерфейс, никакой общей логики
>interface Logistics {
>    Transport createTransport(); // фабричный метод
>}
>
>// Реализации — каждая полностью самостоятельна
>class RoadLogistics implements Logistics {
>    public Transport createTransport() { return new Truck(); }
>    // своя логика доставки, не связанная с другими
>    public void deliver() {
>        System.out.println("Loading on truck...");
>        createTransport().move();
>    }
>}
>
>class AirLogistics implements Logistics {
>    public Transport createTransport() { return new Drone(); }
>    // своя логика доставки, полностью другая
>    public void deliver() {
>        System.out.println("Charging drone...");
>        createTransport().move();
>    }
>}
>```
>
>---
>
>**Когда что выбирать:**
>| | abstract class | interface |
>|---|---|---|
>| Есть общий код для всех | да | нет |
>| Нужны поля/состояние | да | нет |
>| Реализации независимы | нет | да |
>| Современный Java-стиль | — | предпочтительнее |
>
>**Признаки фабричного метода** (независимо от abstract class или interface):
>- возвращает объект (интерфейс или абстрактный класс)
>- подкласс/реализация решает какой конкретный класс создать
>- создатель использует результат, не зная конкретного типа

>[!question]- Сколько фабричных методов может быть? Граница с Abstract Factory
>Фабричных методов может быть несколько — если нужно создавать несколько разных объектов:
>
>```java
>abstract class UIFactory {
>    abstract Button createButton();  // фабричный метод 1
>    abstract Dialog createDialog();  // фабричный метод 2
>
>    void renderForm() {
>        Button b = createButton();
>        Dialog d = createDialog();
>        b.draw();
>        d.show();
>    }
>}
>
>class WindowsUIFactory extends UIFactory {
>    Button createButton() { return new WindowsButton(); }
>    Dialog createDialog() { return new WindowsDialog(); }
>}
>```
>
>**Граница между Factory Method и Abstract Factory:**
>
>| | Factory Method | Abstract Factory |
>|---|---|---|
>| Фабричных методов | обычно один | несколько |
>| Что создаёт | один тип объекта | семейство связанных объектов |
>| Пример | `createTransport()` | `createButton()` + `createDialog()` |
>
>Строгой границы нет — **Abstract Factory это Factory Method, доведённый до создания целого семейства.**

>[!question]- Задача: добавь новый тип в фабрику
>Есть фабрика фигур. Добавь поддержку `Triangle`.
>```java
>interface Shape { double area(); }
>class Circle implements Shape {
>    double r;
>    Circle(double r) { this.r = r; }
>    public double area() { return Math.PI * r * r; }
>}
>class Square implements Shape {
>    double s;
>    Square(double s) { this.s = s; }
>    public double area() { return s * s; }
>}
>class ShapeFactory {
>    public static Shape create(String type, double... args) {
>        return switch (type) {
>            case "circle" -> new Circle(args[0]);
>            case "square" -> new Square(args[0]);
>            // TODO: добавь "triangle" (основание args[0], высота args[1])
>            default -> throw new IllegalArgumentException(type);
>        };
>    }
>}
>```
>**Ответ:**
>```java
>class Triangle implements Shape {
>    double base, height;
>    Triangle(double base, double height) { this.base = base; this.height = height; }
>    public double area() { return 0.5 * base * height; }
>}
>// в switch добавить:
>case "triangle" -> new Triangle(args[0], args[1]);
>```

>[!question]- Abstract Factory — что это и пример на Java
>**Abstract Factory** — порождающий паттерн GoF. Предоставляет интерфейс для создания **семейства связанных объектов**, не указывая их конкретные классы.
>
>**Зачем:** когда нужно создавать несколько объектов, которые должны быть согласованы между собой. Например, UI-компоненты в одном стиле (Windows или Mac — всё сразу, не вперемешку).
>
>**Структура:**
>- `AbstractFactory` — интерфейс с несколькими фабричными методами
>- `ConcreteFactory` — реализует интерфейс, создаёт конкретное семейство
>- Клиент работает только через интерфейс, не знает о конкретных классах
>
>**Пример — UI под разные ОС:**
>```java
>// Продукты
>interface Button { void render(); }
>interface Checkbox { void render(); }
>
>class WindowsButton  implements Button   { public void render() { System.out.println("Windows Button");  } }
>class WindowsCheckbox implements Checkbox { public void render() { System.out.println("Windows Checkbox"); } }
>
>class MacButton   implements Button   { public void render() { System.out.println("Mac Button");   } }
>class MacCheckbox implements Checkbox { public void render() { System.out.println("Mac Checkbox"); } }
>
>// Абстрактная фабрика
>interface UIFactory {
>    Button createButton();
>    Checkbox createCheckbox();
>}
>
>// Конкретные фабрики
>class WindowsFactory implements UIFactory {
>    public Button createButton()     { return new WindowsButton(); }
>    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
>}
>
>class MacFactory implements UIFactory {
>    public Button createButton()     { return new MacButton(); }
>    public Checkbox createCheckbox() { return new MacCheckbox(); }
>}
>
>// Клиент — не знает ни про Windows, ни про Mac
>class Application {
>    private Button button;
>    private Checkbox checkbox;
>
>    Application(UIFactory factory) {
>        button   = factory.createButton();
>        checkbox = factory.createCheckbox();
>    }
>
>    void render() {
>        button.render();
>        checkbox.render();
>    }
>}
>
>// Использование
>UIFactory factory = new WindowsFactory(); // поменяй на MacFactory — всё переключится
>Application app = new Application(factory);
>app.render();
>// Windows Button
>// Windows Checkbox
>```
>
>**Ключевая идея:** меняешь одну фабрику — получаешь полностью другое семейство объектов. Клиентский код не трогаешь.

>[!question]- Задача: Factory Method через наследование
>Реализуй паттерн Factory Method: базовый класс `Notifier` с методом `createMessage()`, подклассы `EmailNotifier` и `SmsNotifier`.
>
>**Ответ:**
>```java
>abstract class Notifier {
>    // Factory Method
>    abstract String createMessage(String text);
>
>    public void send(String text) {
>        String msg = createMessage(text);
>        System.out.println("Sending: " + msg);
>    }
>}
>
>class EmailNotifier extends Notifier {
>    String createMessage(String text) {
>        return "[EMAIL] " + text;
>    }
>}
>
>class SmsNotifier extends Notifier {
>    String createMessage(String text) {
>        return "[SMS] " + text.substring(0, Math.min(text.length(), 160));
>    }
>}
>
>// Использование
>Notifier n = new EmailNotifier();
>n.send("Hello!"); // Sending: [EMAIL] Hello!
>```

>[!question]- Задача: найди проблему в коде с Factory
>Что не так? Как исправить?
>```java
>class ReportFactory {
>    public Report create(String type) {
>        if (type.equals("pdf")) return new PdfReport();
>        if (type.equals("excel")) return new ExcelReport();
>        return null; // <-- проблема?
>    }
>}
>
>Report r = factory.create("xml");
>r.generate(); // NPE!
>```
>**Ответ:**
>Возврат `null` вместо исключения — клиент получает NPE без понятного сообщения.
>
>**Исправление:** бросать `IllegalArgumentException` с описанием:
>```java
>throw new IllegalArgumentException("Unsupported report type: " + type);
>```
>Либо возвращать `Optional<Report>` и явно обрабатывать отсутствие.
