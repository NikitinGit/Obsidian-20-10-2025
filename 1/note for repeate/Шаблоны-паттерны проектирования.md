# Паттерн проектирования Factory
1. [ ] Книга делится на три категории шаблонов: **порождающие**, **структурные** и **поведенческие**. ## Design Patterns: Elements of Reusable Object-Oriented Software

>[!question]- Что такое Factory?
>**Factory (Фабричный метод)** — порождающий паттерн, который определяет интерфейс для создания объектов, но позволяет подклассам решать, какой класс инстанциировать.
>
>**Зачем:** скрыть логику создания объектов, убрать `new ConcreteClass()` из бизнес-кода, легко добавлять новые типы без изменения существующего кода.
>
>**Пример на Java:**
>```java
>// Продукт
>interface Animal {
>    void speak();
>}
>
>class Dog implements Animal {
>    public void speak() { System.out.println("Woof!"); }
>}
>
>class Cat implements Animal {
>    public void speak() { System.out.println("Meow!"); }
>}
>
>// Фабрика
>class AnimalFactory {
>    public static Animal create(String type) {
>        return switch (type) {
>            case "dog" -> new Dog();
>            case "cat" -> new Cat();
>            default -> throw new IllegalArgumentException("Unknown: " + type);
>        };
>    }
>}
>
>// Использование
>Animal a = AnimalFactory.create("dog");
>a.speak(); // Woof!
>```
>
>**Виды:**
>- **Simple Factory** — (не обязательно статический)  метод создаёт объекты (пример выше)
>- **Factory Method** — абстрактный метод в базовом классе, подкласс переопределяет
>- **Abstract Factory** — фабрика фабрик, создаёт семейства связанных объектов

>[!question]- Simple Factory vs Factory Method — отличия
>**Simple Factory** — не паттерн GoF, просто идиома. Один класс со статическим методом и `switch/if` внутри.
>**Factory Method** — официальный паттерн GoF. Абстрактный метод в базовом классе, подкласс решает что создавать.
>
>| | Simple Factory | Factory Method |
>|---|---|---|
>| Паттерн GoF | нет | да |
>| Как добавить новый тип | изменить фабрику | создать новый подкласс |
>| Open/Closed принцип | нарушает | соблюдает |
>| Логика создания | в одном методе | размазана по подклассам |
>| Когда использовать | мало типов, редко меняется | типов много, часто расширяется |
>
>**Simple Factory:**
>```java
>class TransportFactory {
>    public static Transport create(String type) {
>        return switch (type) {
>            case "car"  -> new Car();
>            case "bike" -> new Bike();
>            // добавить Truck? — лезем СЮДА и меняем этот класс
>            default -> throw new IllegalArgumentException(type);
>        };
>    }
>}
>```
>
>**Factory Method:**
>```java
>abstract class Logistics {
>    abstract Transport createTransport(); // фабричный метод
>
>    void deliver() {
>        Transport t = createTransport(); // не знает, что именно
>        t.move();
>    }
>}
>
>class RoadLogistics extends Logistics {
>    Transport createTransport() { return new Truck(); }
>}
>
>class SeaLogistics extends Logistics {
>    Transport createTransport() { return new Ship(); }
>}
>// добавить AirLogistics? — новый класс, старый код не трогаем
>```
>
>**Ключевое отличие одной фразой:**
>Simple Factory — это `if/switch` в одном месте. Factory Method — это полиморфизм вместо `if/switch`.

>[!question]- Три паттерна связанных с фабрикой
>Все три про создание объектов, но на разном уровне сложности:
>
>**1. Simple Factory** — не GoF, просто идиома
>Один класс, один статический метод, `if/switch` внутри. Создаёт один тип объекта.
>
>**2. Factory Method** — GoF, порождающий
>Абстрактный метод в базовом классе. Подкласс переопределяет и решает что создавать. Обычно один фабричный метод.
>
>**3. Abstract Factory** — GoF, порождающий
>Несколько фабричных методов. Создают **семейство связанных объектов** (кнопка + диалог + иконка — всё в одном стиле).
>
>**Эволюция:**
>```
>Simple Factory → Factory Method → Abstract Factory
>  (if/switch)     (1 метод,         (много методов,
>                   полиморфизм)      семейство объектов)
>```
>Чем правее — тем больше гибкости, но и сложности.

>[!question]- Что такое фабричный метод? Нужен ли абстрактный класс?
>**Фабричный метод** — это тот метод, который **создаёт и возвращает объект**.
>Абстрактных методов в классе может быть сколько угодно, фабричным называется именно тот, что отвечает за создание.
>
>**Абстрактный класс — НЕ обязателен.** Важно различать две роли:
>- **Продукт** — то, что создаётся (`interface Transport`) — почти всегда интерфейс
>- **Создатель** — тот, у кого есть фабричный метод — может быть и абстрактным классом, и интерфейсом
>
>---
>
>**Вариант 1: Создатель — абстрактный класс** (есть общая логика)
>
>Используй когда подклассы должны **разделять одинаковое поведение** — оно живёт в базовом классе, переопределять его не нужно.
>
>```java
>// Продукт
>interface Transport {
>    void move();
>}
>class Truck implements Transport {
>    public void move() { System.out.println("Truck driving on road"); }
>}
>class Ship implements Transport {
>    public void move() { System.out.println("Ship sailing on sea"); }
>}
>
>// Создатель — абстрактный класс с ОБЩЕЙ логикой
>abstract class Logistics {
>    abstract Transport createTransport(); // фабричный метод — подкласс обязан реализовать
>
>    // общая логика: оба подкласса используют этот метод БЕЗ переопределения
>    void deliver() {
>        System.out.println("Preparing shipment...");
>        Transport t = createTransport(); // вызывает фабричный метод
>        t.move();
>        System.out.println("Delivered!");
>    }
>}
>
>// Подклассы — реализуют только фабричный метод, deliver() берут из базового класса
>class RoadLogistics extends Logistics {
>    Transport createTransport() { return new Truck(); } // своя реализация
>}
>
>class SeaLogistics extends Logistics {
>    Transport createTransport() { return new Ship(); } // своя реализация
>}
>
>// Использование
>Logistics l = new RoadLogistics();
>l.deliver();
>// Preparing shipment...
>// Truck driving on road
>// Delivered!
>// ^^^ строки "Preparing" и "Delivered" — общая логика из базового класса
>// ^^^ "Truck driving" — результат фабричного метода подкласса
>```
>
>---
>
>**Вариант 2: Создатель — интерфейс** (общей логики нет, каждый сам по себе)
>
>Используй когда подклассы **полностью независимы** и не должны ничего разделять.
>
>```java
>// Продукт
>interface Transport {
>    void move();
>}
>class Truck implements Transport {
>    public void move() { System.out.println("Truck driving"); }
>}
>class Drone implements Transport {
>    public void move() { System.out.println("Drone flying"); }
>}
>
>// Создатель — интерфейс, никакой общей логики
>interface Logistics {
>    Transport createTransport(); // фабричный метод
>}
>
>// Реализации — каждая полностью самостоятельна
>class RoadLogistics implements Logistics {
>    public Transport createTransport() { return new Truck(); }
>    // своя логика доставки, не связанная с другими
>    public void deliver() {
>        System.out.println("Loading on truck...");
>        createTransport().move();
>    }
>}
>
>class AirLogistics implements Logistics {
>    public Transport createTransport() { return new Drone(); }
>    // своя логика доставки, полностью другая
>    public void deliver() {
>        System.out.println("Charging drone...");
>        createTransport().move();
>    }
>}
>```
>
>---
>
>**Когда что выбирать:**
>| | abstract class | interface |
>|---|---|---|
>| Есть общий код для всех | да | нет |
>| Нужны поля/состояние | да | нет |
>| Реализации независимы | нет | да |
>| Современный Java-стиль | — | предпочтительнее |
>
>**Признаки фабричного метода** (независимо от abstract class или interface):
>- возвращает объект (интерфейс или абстрактный класс)
>- подкласс/реализация решает какой конкретный класс создать
>- создатель использует результат, не зная конкретного типа

>[!question]- Сколько фабричных методов может быть? Граница с Abstract Factory
>Фабричных методов может быть несколько — если нужно создавать несколько разных объектов:
>
>```java
>abstract class UIFactory {
>    abstract Button createButton();  // фабричный метод 1
>    abstract Dialog createDialog();  // фабричный метод 2
>
>    void renderForm() {
>        Button b = createButton();
>        Dialog d = createDialog();
>        b.draw();
>        d.show();
>    }
>}
>
>class WindowsUIFactory extends UIFactory {
>    Button createButton() { return new WindowsButton(); }
>    Dialog createDialog() { return new WindowsDialog(); }
>}
>```
>
>**Граница между Factory Method и Abstract Factory:**
>
>| | Factory Method | Abstract Factory |
>|---|---|---|
>| Фабричных методов | обычно один | несколько |
>| Что создаёт | один тип объекта | семейство связанных объектов |
>| Пример | `createTransport()` | `createButton()` + `createDialog()` |
>
>Строгой границы нет — **Abstract Factory это Factory Method, доведённый до создания целого семейства.**

>[!question]- Задача: добавь новый тип в фабрику
>Есть фабрика фигур. Добавь поддержку `Triangle`.
>```java
>interface Shape { double area(); }
>class Circle implements Shape {
>    double r;
>    Circle(double r) { this.r = r; }
>    public double area() { return Math.PI * r * r; }
>}
>class Square implements Shape {
>    double s;
>    Square(double s) { this.s = s; }
>    public double area() { return s * s; }
>}
>class ShapeFactory {
>    public static Shape create(String type, double... args) {
>        return switch (type) {
>            case "circle" -> new Circle(args[0]);
>            case "square" -> new Square(args[0]);
>            // TODO: добавь "triangle" (основание args[0], высота args[1])
>            default -> throw new IllegalArgumentException(type);
>        };
>    }
>}
>```
>**Ответ:**
>```java
>class Triangle implements Shape {
>    double base, height;
>    Triangle(double base, double height) { this.base = base; this.height = height; }
>    public double area() { return 0.5 * base * height; }
>}
>// в switch добавить:
>case "triangle" -> new Triangle(args[0], args[1]);
>```

>[!question]- Abstract Factory — что это и пример на Java
>**Abstract Factory** — порождающий паттерн GoF. Предоставляет интерфейс для создания **семейства связанных объектов**, не указывая их конкретные классы.
>
>**Зачем:** когда нужно создавать несколько объектов, которые должны быть согласованы между собой. Например, UI-компоненты в одном стиле (Windows или Mac — всё сразу, не вперемешку).
>
>**Структура:**
>- `AbstractFactory` — интерфейс с несколькими фабричными методами
>- `ConcreteFactory` — реализует интерфейс, создаёт конкретное семейство
>- Клиент работает только через интерфейс, не знает о конкретных классах
>
>**Пример — UI под разные ОС:**
>```java
>// Продукты
>interface Button { void render(); }
>interface Checkbox { void render(); }
>
>class WindowsButton  implements Button   { public void render() { System.out.println("Windows Button");  } }
>class WindowsCheckbox implements Checkbox { public void render() { System.out.println("Windows Checkbox"); } }
>
>class MacButton   implements Button   { public void render() { System.out.println("Mac Button");   } }
>class MacCheckbox implements Checkbox { public void render() { System.out.println("Mac Checkbox"); } }
>
>// Абстрактная фабрика
>interface UIFactory {
>    Button createButton();
>    Checkbox createCheckbox();
>}
>
>// Конкретные фабрики
>class WindowsFactory implements UIFactory {
>    public Button createButton()     { return new WindowsButton(); }
>    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
>}
>
>class MacFactory implements UIFactory {
>    public Button createButton()     { return new MacButton(); }
>    public Checkbox createCheckbox() { return new MacCheckbox(); }
>}
>
>// Клиент — не знает ни про Windows, ни про Mac
>class Application {
>    private Button button;
>    private Checkbox checkbox;
>
>    Application(UIFactory factory) {
>        button   = factory.createButton();
>        checkbox = factory.createCheckbox();
>    }
>
>    void render() {
>        button.render();
>        checkbox.render();
>    }
>}
>
>// Использование
>UIFactory factory = new WindowsFactory(); // поменяй на MacFactory — всё переключится
>Application app = new Application(factory);
>app.render();
>// Windows Button
>// Windows Checkbox
>```
>
>**Ключевая идея:** меняешь одну фабрику — получаешь полностью другое семейство объектов. Клиентский код не трогаешь.

>[!question]- Задача: Factory Method через наследование
>Реализуй паттерн Factory Method: базовый класс `Notifier` с методом `createMessage()`, подклассы `EmailNotifier` и `SmsNotifier`.
>
>**Ответ:**
>```java
>abstract class Notifier {
>    // Factory Method
>    abstract String createMessage(String text);
>
>    public void send(String text) {
>        String msg = createMessage(text);
>        System.out.println("Sending: " + msg);
>    }
>}
>
>class EmailNotifier extends Notifier {
>    String createMessage(String text) {
>        return "[EMAIL] " + text;
>    }
>}
>
>class SmsNotifier extends Notifier {
>    String createMessage(String text) {
>        return "[SMS] " + text.substring(0, Math.min(text.length(), 160));
>    }
>}
>
>// Использование
>Notifier n = new EmailNotifier();
>n.send("Hello!"); // Sending: [EMAIL] Hello!
>```

>[!question]- Abstract Factory: interface или abstract class? (ответ для собеседования)
>В Abstract Factory **обычно интерфейс**, в отличие от Factory Method, где чаще abstract class.
>
>**Почему — главный критерий выбора:**
>
>| Есть ли общий код или состояние у потомков? | Что брать |
>|---|---|
>| Да (поля, готовые методы, template method) | `abstract class` |
>| Нет (только контракт) | `interface` |
>
>**Применительно к двум паттернам:**
>
>| | Factory Method | Abstract Factory |
>|---|---|---|
>| Что нужно от базового типа | абстрактный метод **+ общая логика** (`deliver()`, поле `region`) | **только контракт** — список методов создания |
>| Есть ли что-то общее у наследников | да — общий алгоритм работы | нет — каждая фабрика просто создаёт своё семейство |
>| Что выбрать | abstract class | interface |
>
>В Factory Method есть template-метод (`deliver()`), который вызывает фабричный метод — этот алгоритм надо где-то хранить, интерфейс не подходит.
>В Abstract Factory **ничего общего** между конкретными фабриками нет — обе просто реализуют контракт. Делить нечего → достаточно интерфейса.
>
>---
>
>**Готовый ответ для собеседования:**
>
>> «В Abstract Factory у конкретных фабрик нет общего кода и общего состояния — каждая просто реализует свой контракт создания семейства продуктов. Делить им нечего, поэтому достаточно интерфейса. Абстрактный класс был бы оправдан, только если бы у фабрик появилось общее поведение или поля.
>>
>> Это отличает Abstract Factory от Factory Method, где в базовом классе обычно живёт template-метод вроде `deliver()`, использующий фабричный метод — там без абстрактного класса не обойтись.
>>
>> Дополнительный плюс интерфейса: конкретная фабрика может одновременно наследовать какой-то базовый класс (для логирования, DI) и реализовывать интерфейс фабрики. С abstract class так не получится — Java не разрешает множественное наследование классов».
>
>---
>
>**❌ Чего НЕ говорить:**
>- «слишком высокий уровень абстракции» — расплывчато, непонятно, звучит как заученное
>- «потому что множественное наследование запрещено» — это аргумент за интерфейсы **вообще**, а не за Abstract Factory конкретно. К Factory Method тот же аргумент применим, но там как раз чаще abstract class. То есть аргумент не объясняет **разницу** между паттернами.
>
>---
>
>**Когда в Abstract Factory всё-таки взять abstract class:**
>Если у конкретных фабрик появилось общее состояние или общая логика:
>```java
>public abstract class TransportOSFactory {
>    protected final String version;          // общее поле
>
>    protected TransportOSFactory(String v) { this.version = v; }
>
>    public abstract Car   createCar();
>    public abstract Ship  createShip();
>    public abstract Truck createTruck();
>
>    public void printInfo() {                // общий метод
>        System.out.println("Factory version: " + version);
>    }
>}
>```
>Без этого — `interface` уместнее.
>
>---
>
>**Главная мысль одной фразой:**
>Выбор между `interface` и `abstract class` всегда упирается в один вопрос: **есть ли что-то общее, что хочется вынести в базовый тип?** Да → abstract class. Нет → interface.

# Паттерн проектирования Prototype (Прототип)

>[!question]- Что такое Prototype и зачем нужен?
>**Prototype (Прототип)** — порождающий паттерн GoF. Позволяет копировать существующие объекты, **не зная их конкретного класса**. Клиент работает с интерфейсом — вызывает у него `clone()`, получает копию того же типа.
>
>**Зачем:**
>- избавиться от `new ConcreteClass()` в клиентском коде
>- копировать сложные объекты, не разбираясь в их внутренней структуре
>- создавать варианты объекта на основе "эталона" (заготовки)
>- избежать дорогой инициализации — если объект создаётся долго (тяжёлые вычисления, БД, файлы), проще клонировать готовый
>
>**Когда применять:**
>- Объект сложно собрать с нуля, но легко скопировать
>- Нужно создавать много похожих объектов с небольшими отличиями
>- Хочется уйти от иерархии классов фабрик — иногда Prototype проще, чем Abstract Factory
>
>**Структура:**
>- `Prototype` — интерфейс с методом `clone()`
>- `ConcretePrototype` — реализует `clone()`, обычно через конструктор копирования
>- (опционально) `PrototypeRegistry` — хранит "эталоны", отдаёт копии по ключу

>[!question]- Prototype на Java — современный пример
>В современной Java **не используют `Cloneable`/`Object.clone()`** — это устаревший механизм с кучей подводных камней (защищённый метод, `CloneNotSupportedException`, shallow copy по умолчанию). Идиоматичный способ — собственный интерфейс с `clone()` + конструктор копирования.
>
>**Интерфейс:**
>```java
>public interface Shape {
>    Shape clone();          // СВОЙ метод, не java.lang.Object#clone
>    void draw();
>}
>```
>
>**Конкретный прототип:**
>```java
>public class Circle implements Shape {
>    private int radius;
>    private String color;
>
>    public Circle(int radius, String color) {
>        this.radius = radius;
>        this.color = color;
>    }
>
>    // конструктор копирования — ядро Prototype в Java
>    private Circle(Circle source) {
>        this.radius = source.radius;
>        this.color = source.color;
>    }
>
>    @Override
>    public Shape clone() {
>        return new Circle(this);   // делегируем конструктору копирования
>    }
>
>    @Override
>    public void draw() {
>        System.out.println("Circle r=" + radius + ", color=" + color);
>    }
>}
>```
>
>**Использование:**
>```java
>Shape original = new Circle(10, "red");
>Shape copy = original.clone();          // клиент НЕ знает, что это Circle
>
>original.draw();    // Circle r=10, color=red
>copy.draw();        // Circle r=10, color=red
>System.out.println(original == copy);   // false — это два разных объекта
>```
>
>**Ключевая идея:** клиент работает только с `Shape`. Он не знает, что внутри `Circle` или `Rectangle`. Метод `clone()` сам разбирается, как скопировать своё состояние.

>[!question]- Prototype Registry — реестр прототипов
>Типичное расширение паттерна. Хранит "эталонные" заготовки в `Map`, клиенту отдаёт **копию** по ключу. Оригинал остаётся в реестре нетронутым.
>
>```java
>public class ShapeRegistry {
>    private final Map<String, Shape> prototypes = new HashMap<>();
>
>    public void register(String key, Shape prototype) {
>        prototypes.put(key, prototype);
>    }
>
>    public Shape get(String key) {
>        Shape prototype = prototypes.get(key);
>        if (prototype == null) {
>            throw new IllegalArgumentException("Unknown: " + key);
>        }
>        return prototype.clone();   // ВАЖНО: отдаём копию, не эталон
>    }
>}
>
>// Использование:
>ShapeRegistry registry = new ShapeRegistry();
>registry.register("red-circle", new Circle(5, "red"));
>registry.register("blue-rect",  new Rectangle(4, 8, "blue"));
>
>Shape s1 = registry.get("red-circle");
>Shape s2 = registry.get("red-circle");  // отдельная копия!
>System.out.println(s1 == s2);            // false
>```
>
>**Зачем реестр:**
>- Убирает `if/switch` из клиентского кода — реестр сам знает, что клонировать
>- "Эталон" настраивается один раз (например, при старте приложения), дальше — только копии
>- Можно подменять прототипы во время выполнения, без изменения кода клиента

>[!question]- Shallow copy vs Deep copy (важный нюанс Prototype)
>**Shallow copy (поверхностная):** копируются поля как есть. Примитивы и `String` копируются по значению, ссылки на объекты — **по ссылке** (обе копии указывают на тот же вложенный объект).
>
>**Deep copy (глубокая):** копируются и сам объект, и все вложенные объекты рекурсивно.
>
>```java
>public class Order {
>    private String name;
>    private List<String> items;   // <-- ссылочное поле
>
>    // SHALLOW — items указывает на тот же List
>    private Order(Order source) {
>        this.name = source.name;
>        this.items = source.items;          // ❌ обе копии шарят список
>    }
>
>    // DEEP — items копируется в новый List
>    private Order(Order source) {
>        this.name = source.name;
>        this.items = new ArrayList<>(source.items);   // ✅ свой список
>    }
>}
>```
>
>**Почему важно:** если вложенный объект **мутабелен** (`ArrayList`, `Date`, кастомный класс), то shallow copy создаёт скрытую связь между копиями — изменив поле в одной, сломаешь другую. Для **иммутабельных** вложенных полей (`String`, `BigDecimal`, рекорды без ссылок) shallow copy безопасна.
>
>**Правило:** если в объекте есть мутабельные коллекции или объекты — копируй их явно (deep copy).

>[!question]- Prototype vs Factory — в чём разница и когда что
>Оба паттерна создают объекты, но по-разному:
>
>| | Factory | Prototype |
>|---|---|---|
>| Создаёт объект | "с нуля" через `new` | копированием готового |
>| Что нужно для создания | параметры (тип, аргументы) | существующий экземпляр |
>| Знание конкретных классов | у фабрики — да (`new Circle()`) | нигде — клонирование через интерфейс |
>| Дорогая инициализация | каждый раз заново | один раз, потом только копии |
>| Когда использовать | типов мало, создание дешёвое | типов много, создание дорогое или конфигурация сложная |
>
>**Пример где Prototype лучше Factory:**
>Если объект собирается долго (запросы в БД, расчёты, чтение файла), но дальше нужны 1000 похожих экземпляров — один раз собрали "эталон", дальше клонируем.
>
>**Пример где Factory лучше Prototype:**
>Если у объекта простая инициализация, но нужно гибко выбирать тип — `Factory.create("circle")` понятнее, чем "сначала зарегистрируй circle в реестре, потом проси оттуда".
>
>Иногда комбинируют: фабрика **внутри** использует реестр прототипов.

>[!question]- Подводные камни Prototype
>**1. Java `Cloneable` — не использовать.**
>Стандартный `java.lang.Cloneable` + `Object.clone()` — устаревший механизм:
>- `clone()` объявлен `protected` в `Object` — нужно переопределять `public`
>- Бросает checked `CloneNotSupportedException` — нужно ловить или объявлять
>- По умолчанию shallow copy
>- `Cloneable` — маркерный интерфейс **без** метода (странный дизайн)
>
>Современная Java делает свой интерфейс с `clone()` + конструктор копирования (как в примере выше).
>
>**2. Deep copy сложен для глубоких иерархий.**
>Если в объекте есть мутабельные коллекции с мутабельными объектами внутри — копировать всё руками громоздко. Иногда используют сериализацию (через `Serializable` или JSON) — медленно, но универсально.
>
>**3. Циклические ссылки.**
>Если в графе объектов есть цикл (A → B → A), наивное копирование зациклится. Решение — `IdentityHashMap` "оригинал → копия" внутри метода клонирования.
>
>**4. final-поля.**
>В конструкторе копирования final-поля присваиваются нормально (это конструктор!). А вот если бы пытались собирать копию через сеттеры — final-поля не дали бы.

>[!question]- Где Prototype встречается в реальном коде
>- **`String.toCharArray()`** — фактически клонирование внутреннего массива
>- **`ArrayList(Collection)`** — конструктор копирования = Prototype через `new ArrayList<>(other)`
>- **`Object#clone()`** — устаревший вариант, но всё ещё встречается в legacy
>- **Spring `@Scope("prototype")`** — name похожее, **но это другое**! Это про скоуп бина (каждый раз новый объект), а не про паттерн копирования
>- **Java Records `with`-методы** (через библиотеки или вручную) — концептуально близко: "копия с одним изменённым полем"
>- **Builder + копирование** — `Builder.from(existing).withX(...).build()` — комбинация Builder и Prototype

>[!question]- Задача: реализуй Prototype для класса User
>Дан класс. Реализуй копирование через паттерн Prototype.
>```java
>public class User {
>    private String name;
>    private int age;
>    private List<String> roles;
>    // ...
>}
>```
>**Ответ:**
>```java
>public interface CloneableUser {
>    CloneableUser clone();
>}
>
>public class User implements CloneableUser {
>    private String name;
>    private int age;
>    private List<String> roles;
>
>    public User(String name, int age, List<String> roles) {
>        this.name = name;
>        this.age = age;
>        this.roles = new ArrayList<>(roles);  // deep copy на входе тоже полезно
>    }
>
>    private User(User src) {
>        this.name = src.name;            // String иммутабелен — shallow OK
>        this.age = src.age;              // примитив — копия по значению
>        this.roles = new ArrayList<>(src.roles);   // мутабельный List — DEEP copy!
>    }
>
>    @Override
>    public CloneableUser clone() {
>        return new User(this);
>    }
>}
>```
>**Что важно сказать на собеседовании:**
>- Не использовать `Cloneable`/`Object.clone()` — устаревший механизм
>- Внутри — конструктор копирования
>- Для `List<String>` обязательно создавать новый список (deep copy), иначе копии будут шарить состояние
>- `String` копировать не нужно — он иммутабелен

>[!question]- Задача: найди проблему в коде с Factory
>Что не так? Как исправить?
>```java
>class ReportFactory {

>        if (type.equals("excel")) return new ExcelReport();
