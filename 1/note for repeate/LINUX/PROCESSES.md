# Процессы, потоки, shell, память

>[!question]- shell (оболочка) это
> программа-посредник между тобой и ядром ОС. Читает команды, которые ты печатаешь в терминале (`ls`, `cd`, `cat`), разбирает их (имя команды, аргументы, перенаправления `>`, `|`, `2>&1`) и просит ядро запустить нужную программу.
> Примеры shell: `bash`, `zsh` (стоит у меня), `sh`, `fish`.
> Важно: `shell` — это обычная программа/процесс, он **ничем не управляет** в плане многозадачности. Синтаксис `|`, `2>&1`, `&&` понимает и обрабатывает именно shell, а не сама команда (`ls`).

>[!question]- ядро ОС (kernel) это
> сердце операционной системы (Linux). Управляет процессами, потоками, памятью, CPU, файловыми дескрипторами; распределяет процессы по ядрам процессора.
> ВАЖНО: ядро ОС ≠ shell. shell — это просто программа под управлением ядра. Параллельностью процессов и потоков управляет **ядро**, а не shell.

>[!question]- fork() это
> системный вызов (запрос к ядру), который создаёт новый процесс — почти точную копию текущего. Появляются два процесса: родитель (parent) и потомок (child, копия). Копируется почти всё, включая таблицу файловых дескрипторов (0,1,2).
> Название от англ. fork — "развилка": один поток исполнения раздваивается.

>[!question]- exec() это
> семейство системных вызовов, которое заменяет программу внутри текущего процесса на другую. Процесс остаётся тем же (тот же PID, та же таблица дескрипторов), но его код полностью замещается новой программой.
> Важно: `exec()` НЕ создаёт новый процесс — он перезаписывает существующий. Возврата к старому коду уже не будет.

>[!question]- паттерн fork + exec  (как запускается любая внешняя команда)
> ```
> 1. shell делает fork()      → появляется процесс-потомок (копия shell)
> 2. потомок (опц.) настраивает перенаправления (2>&1, >file, |)
> 3. потомок делает exec("ls") → превращается в программу ls
> 4. родитель-shell ждёт (wait), пока ls завершится
> 5. ls закончил → shell снова готов принимать команды
> ```
> Зачем 2 шага, а не один "запусти программу": ИМЕННО между fork и exec потомок может настроить окружение (переназначить дескрипторы `2>&1`, сменить директорию, права) — не трогая родителя. Поэтому перенаправления в shell вообще возможны.

>[!question]- модель fork+exec — это только Unix или Windows тоже? какие ОС по ней работают
> Надо разделить 2 вещи:
> 1. ОТНОШЕНИЕ родитель/потомок (дерево процессов) — есть почти во ВСЕХ ОС, включая Windows. У процессов везде есть родитель и потомки.
> 2. МЕХАНИЗМ `fork`+`exec` (склонировать себя, потом заменить программу) — это наследие UNIX, есть только в Unix-семействе.
> По fork+exec работают: Linux (все дистрибутивы), macOS (ядро BSD/Mach), Android (ядро Linux; приложения через процесс zygote — это тоже fork), iOS, BSD (FreeBSD/OpenBSD), Solaris, AIX, HP-UX.
> Windows работает ИНАЧЕ — один системный вызов `CreateProcess()`, который создаёт процесс И сразу загружает программу за ОДИН шаг. Промежуточной "копии родителя" нет.
> ```
> Unix:     fork() --> [копия родителя] --> exec("prog") --> [prog]   (2 шага)
> Windows:  CreateProcess("prog.exe", ...) --> [сразу prog]            (1 шаг)
> ```
> | | Unix (fork+exec) | Windows (CreateProcess) |
> |---|---|---|
> | Шагов на запуск | 2 | 1 |
> | Копия родителя | есть | нет |
> | Настройка окружения потомка | в коде МЕЖДУ fork и exec | параметрами вызова (STARTUPINFO) |
> | Перенаправление `2>&1` | потомок сам правит дескрипторы перед exec | через параметры CreateProcess |
> Перенаправления `2>&1`/`>file` в Unix возможны именно потому, что есть "окно" между fork и exec, где потомок настраивает себя. В Windows такого окна нет.
> Факт: в Windows есть WSL и подсистема POSIX, которые ЭМУЛИРУЮТ `fork()` для Unix-программ — реализовать честный fork поверх Windows-ядра технически тяжело, т.к. нативная модель совсем другая.

>[!question]- процесс (process) vs поток (thread)
> | | Процесс | Поток (thread) |
> |---|---|---|
> | Память | своя, изолированная | общая с другими потоками процесса |
> | Создаётся | `fork()` / `fork`+`exec` | внутри программы (например JVM) |
> | Пример | shell, javac, ls | потоки внутри JVM |
> | Свой PID | да | нет (только ID потока внутри процесса) |
> Иерархия: ядро ОС → процессы (свой PID, своя память) → потоки внутри процесса (делят память процесса).

>[!question]- что является процессом-потомком, а что нет (ls/grep vs cd)
> Внешняя команда = отдельная программа на диске (есть исполняемый файл). shell запускает её через `fork`+`exec` → это ОТДЕЛЬНЫЙ процесс-потомок со своим PID. Примеры: `ls`, `grep`, `cat`, `javac`, `cp`, `mv`, `python`.
> Встроенная команда (builtin) = часть самого shell. НЕ порождает новый процесс (нет fork/exec). Примеры: `cd`, `pwd`, `echo` (в shell), `export`, `alias`.
> Проверить тип: команда `type ls` / `type cd`.

>[!question]- почему cd — это builtin, а не отдельная программа/потомок
> `cd` должна менять текущую директорию САМОГО shell. Если бы `cd` запускался в потомке (через fork), он сменил бы директорию только у копии-потомка, которая тут же умрёт, а родительский shell остался бы на месте. Поэтому `cd` встроен прямо в shell.
> Для примера "новый процесс с тремя дескрипторами" всегда бери внешнюю команду — `ls`, `cat`, `grep`, а не `cd`.

>[!question]- зачем родителю нужен процесс-потомок (зачем вообще fork)
> 1. ГЛАВНОЕ — запустить чужую программу, не убив себя. `exec()` затирает текущий код. Если бы shell сделал exec без fork, он бы перестал существовать (его код затёрт), и терминал выполнил бы ровно одну команду за всю жизнь. Поэтому жертвуют потомком-двойником, а родитель остаётся жив.
> 2. Изоляция — у потомка своя память; если он упадёт, родитель не пострадает.
> 3. Результат — родитель через `wait()` забирает код завершения потомка (`$?`: 0 = успех). На этом основан `&&`.
> 4. Параллельность — можно не ждать потомка (`javac ... &`) и работать дальше; так работают фоновые задачи и серверы.
> Все процессы образуют ДЕРЕВО с корнем в `init`/`systemd` (PID 1).

>[!question]- shell привязан к директории? (CWD)
> Да. У каждого процесса есть атрибут — текущая рабочая директория (CWD, current working directory). Это часть состояния процесса. `cd` меняет CWD именно процесса shell. Относительные пути (`./file`, `../`) считаются от неё. Посмотреть: `pwd`.

>[!question]- 2 команды в 2 вкладках терминала одновременно — будут работать параллельно?
> Да. Каждая вкладка/окно терминала запускает СВОЙ отдельный процесс shell (свой PID). Это независимые процессы со своим CWD, своими переменными окружения, своей историей.
> Параллельность определяется тем, что это РАЗНЫЕ процессы (разные PID), а НЕ тем, в какой они директории. Одна общая директория не мешает — это просто общий атрибут, а не блокировка.
> `echo $$` выводит PID текущего shell — в двух вкладках числа будут разные.
> Общая директория может дать эффект, только если обе команды одновременно пишут в ОДИН файл (race condition) — но это про общий файл, а не про директорию.

>[!question]- файловые дескрипторы 0 / 1 / 2 (stdin/stdout/stderr)
> 0,1,2 — это не объекты, а целые числа: индексы в таблице открытых файлов процесса, которую ведёт ядро.
> 0 → stdin (ввод), 1 → stdout (обычный вывод), 2 → stderr (ошибки).
> За каждым числом — ссылка на источник/приёмник байтов (терминал, файл, pipe, сокет, /dev/null и др. — список открытый, "всё есть файл"). Дескрипторы можно переназначать (`2>&1`, `>file`, `<file`). Дескрипторов может быть больше трёх: 3,4,5… процесс открывает сам, открывая файлы/сокеты.
> При завершении процесса ядро закрывает его дескрипторы. Но сам объект (терминал, файл) живёт, пока на него ссылается ХОТЬ КТО-ТО (ядро ведёт счётчик ссылок). См. [[FLAGS]] про `2>&1`.

>[!question]- дескриптор (Unix) vs поток-объект Stream (Java) — это одно и то же?
> Нет, это разные уровни одного и того же.
> Unix-дескриптор (0,1,2) — целое число, абстракция уровня ЯДРА ОС.
> Java `System.out`/`System.in`/`System.err` — это ОБЪЕКТЫ классов, абстракция уровня языка/JVM, обёртки поверх тех самых дескрипторов.
> `System.out.println("hi")` в итоге вызывает системный вызов `write(1, ...)` — то есть пишет в дескриптор 1.
> Аналогия: дескриптор 1 — розетка в стене; `System.out` — удлинитель, воткнутый в неё; `2>&1` — электрик, переподключающий провода в самой стене (на уровне ОС, до запуска программы), не трогая удлинители.

>[!question]- изоляция памяти процессов — это про что
> "У каждого процесса своя память" = ИЗОЛЯЦИЯ АДРЕСНЫХ ПРОСТРАНСТВ: процесс A не может прочитать или испортить память процесса B. Ядро через таблицы страниц переводит виртуальные адреса в физическую RAM.
> ВАЖНО: изоляция отвечает на вопрос "кто к чьей памяти имеет доступ", а НЕ "сколько физической RAM каждому достанется".
> | | Изоляция | Квота (лимит) |
> |---|---|---|
> | Что | A не лезет в память B | A не может занять больше N байт |
> | По умолчанию | есть всегда | НЕТ, лимита нет |
> Физическая RAM — ОБЩИЙ ресурс на всех. По умолчанию у процесса нет потолка — он может просить память, пока не кончится общая RAM.

>[!question]- почему при OOM в java зависает вся система, если у процесса своя память
> Изоляция защищает от порчи чужих данных, но НЕ даёт гарантированный кусок RAM — физическая память общая. Бывает 2 разных вида "Java OOM":
> 1. `java.lang.OutOfMemoryError` (упёрлись в лимит кучи JVM `-Xmx`) — падает ТОЛЬКО процесс java, система в порядке. Тут изоляция работает.
> 2. Кончилась вся физическая RAM+swap машины (java запущена без вменяемого `-Xmx`) — вот это вешает всё:
>    - SWAP THRASHING (пробуксовка): RAM полна → ядро выгружает страницы на диск → они тут же снова нужны → грузит обратно → по кругу. Диск в тысячи раз медленнее RAM, поэтому 99% времени CPU уходит на возню RAM↔диск. Система не упала, а захлёбывается — снаружи это "зависание". Страдают ВСЕ процессы, т.к. замедлился общий ресурс.
>    - OOM killer: компонент ядра выбирает процесс-жертву (обычно самый прожорливый — ту самую java) и убивает, чтобы спасти систему.
> Аналогия: в доме у каждого своя квартира (изоляция), но электричество/вода/лифт — общие. Один жилец перегрузил сеть → вырубило весь дом, хотя в чужие квартиры никто не входил.

>[!question]- как сделать, чтобы один процесс не вешал всю систему (лимиты памяти)
> Лимита по умолчанию нет — его надо назначить:
> - `-Xmx` у JVM (напр. `-Xmx512m`) — ограничить кучу Java; тогда упрётся java (вид 1), а не вся система.
> - `ulimit -v` — лимит виртуальной памяти на процесс в shell.
> - cgroups (systemd, Docker, Kubernetes) — ядро жёстко ограничивает группе процессов RAM; превысила — убьют её, а не всех. Так контейнеры изолируют ресурсы.
> - `earlyoom` / `systemd-oomd` — раньше зовут OOM killer, не давая системе уйти в долгую пробуксовку.
