>[!question]- 2>&1  это
>   перенаправление потоков вывода. У программы есть три стандартных потока: `0` - stdin (ввод), `1` - stdout (обычный вывод), `2` - stderr (вывод ошибок).
>   `2>&1` означает "перенаправь поток `2` (ошибки) туда же, куда идёт поток `1` (обычный вывод)". То есть ошибки сольются с обычным выводом в один поток.
>   Пример
>   ```
>   ls -la "/путь/которого/нет/" 2>&1 | head -50
>   ```
>   без `2>&1` ошибка `ls` ушла бы на экран отдельно и не попала бы в pipe (`|` передаёт дальше только stdout). С `2>&1` ошибка тоже пройдёт через конвейер.
>   ВАЖНО: если команда отработала УСПЕШНО (папка есть, ошибок нет) — stderr пустой, и `2>&1` НИЧЕГО не меняет, вывод одинаковый. Разница видна ТОЛЬКО когда есть ошибка. Проверять эффект `2>&1` надо на команде, которая реально выдаёт ошибку (напр. несуществующая папка).
>   Наглядный пример с `/dev/null` (это "мусорка" — туда выкидывается всё, что придёт). Ставим `> /dev/null` в конец, чтобы выкинуть то, что прошло через pipe:
>   ```
>   # БЕЗ 2>&1 — ошибка летит МИМО pipe, сразу на ЭКРАН (хотя финал в /dev/null)
>   ls -la "/НЕТ_ПАПКИ/" | head -5 > /dev/null
>   → на экране: "/НЕТ_ПАПКИ/": No such file or directory
>
>   # С 2>&1 — ошибка идёт В pipe → head → /dev/null, на экране ПУСТО
>   ls -la "/НЕТ_ПАПКИ/" 2>&1 | head -5 > /dev/null
>   → на экране ничего (ошибку поглотил /dev/null через pipe)
>   ```
>   Схема:
>   ```
>   без 2>&1:  ls --stdout(пусто)--> head --> /dev/null
>              ls --stderr(ошибка)----------> ЭКРАН   ← мимо pipe!
>
>   с 2>&1:    ls --stdout─┐
>              ls --stderr─┴(слились)--> head --> /dev/null  (всё выкинуто, экран пуст)
>   ```
>   Вывод: `2>&1` нужен, чтобы ошибки ТОЖЕ проходили через pipe вместе с обычным выводом (можно отфильтровать через grep, обрезать head, записать в файл, выкинуть в /dev/null), а не вылетали отдельно на экран.

>[!question]- head -N  это
>   показывает только первые N строк из того, что получит на вход, остальное отбрасывает. Часто стоит после pipe `|`, чтобы не вывалить на экран весь длинный вывод.
>   Пример
>   ```
>   ls -la 2>&1 | head -50
>   ```
>   покажет только первые 50 строк списка файлов.
>   Противоположность - `tail -N`, которая показывает последние N строк.

>[!question]- /dev/null  и конструкции перенаправления (2>/dev/null, >/dev/null 2>&1)
>   `/dev/null` — это "мусорка": специальное устройство, всё записанное в него просто исчезает. Используется, чтобы СПРЯТАТЬ (выкинуть) ненужный вывод.
>   ВАЖНО: `2>&1` сам по себе ничего не прячет — он лишь СЛИВАЕТ ошибки (stderr, поток 2) в общий поток с обычным выводом (stdout, поток 1), чтобы обрабатывать их вместе (отфильтровать grep'ом, сохранить в файл, пролистать). Прячет — именно `/dev/null`.
>   Таблица конструкций:
>   ```
>   команда 2>/dev/null        # спрятать ТОЛЬКО ошибки (обычный вывод показать)
>   команда >/dev/null         # спрятать ТОЛЬКО обычный вывод (ошибки показать)
>   команда >/dev/null 2>&1    # спрятать ВСЁ (и вывод, и ошибки) — полная тишина
>   команда >file 2>&1         # сохранить в файл И вывод, И ошибки
>   команда 2>&1 | grep error  # отфильтровать ошибки grep'ом (без 2>&1 grep их не увидит)
>   ```
>   Как читается `>/dev/null 2>&1`: сначала `>/dev/null` отправляет stdout (1) в мусорку, затем `2>&1` досылает stderr (2) туда же, куда уже идёт поток 1 → молчит всё.
>   Частый пример (есть в [[BASE]]): `find / -type d -name ".obsidian" 2>/dev/null` — `2>/dev/null` прячет ошибки `Permission denied`, а найденные папки (поток 1) показывает.
>   Не путать: `2>/dev/null` (ошибки В мусорку = спрятать) vs `2>&1` (ошибки К обычному выводу = вести вместе).
