# Ошибки и решения БД

## Lock wait timeout exceeded (MySQL 1205 / SQLState 40001)

### Симптом
В логах бэкенда сыплется:
```
SQL Error: 1205, SQLState: 40001
Lock wait timeout exceeded; try restarting transaction
org.springframework.dao.PessimisticLockingFailureException
```
Сайт «не грузится», запросы висят и отваливаются через ~50 сек (`innodb_lock_wait_timeout`).

### Причина
Кто-то держит блокировку строк (row lock) в незакоммиченной транзакции, остальные стоят в очереди.

Частый кейс: **дебаг по живой (preprod/общей) БД**. Поставил брейкпоинт внутри `@Transactional`-метода, постоял на нём 15 минут — транзакция всё это время открыта и держит локи. Параллельные запросы (например `touchToken` по `user_tokens`, дёргается на каждый запрос фронта) упираются в неё и падают с 1205.

⚠️ Важный нюанс: если прибить локальное приложение **жёстко**, TCP-коннект к удалённому MySQL может не закрыться корректно. Сервер не знает, что клиент умер, и держит сессию + открытую транзакцию + локи до `wait_timeout` (по умолчанию **8 часов**). Поэтому перезапуск приложения/докера НЕ помогает — мёртвый коннект «осиротел» на стороне сервера, убивать надо его там.

### Диагностика (от быстрого к детальному)

**1. Логи бэкенда — первый сигнал:**
```bash
docker logs --tail 200 <backend> | grep -iE "1205|lock wait|40001|deadlock"
```
- `1205 / Lock wait timeout` → кто-то держит лок.
- `1213 / Deadlock` → взаимоблокировка, InnoDB разрулит сам (откатит одну транзакцию).

**2. Кто что делает:**
```sql
SHOW PROCESSLIST;
```
На что смотреть:
- `Command = Query` + большой `Time` → запрос реально выполняется долго / ждёт лок.
- `Command = Sleep` + большой `Time` → открытая транзакция, ничего не делает, но **держит локи** (осиротевший дебаг).
- много коннектов в `State = ... lock` → очередь за одной строкой.

С привилегией `PROCESS` можно отфильтровать:
```sql
SELECT id, user, host, db, command, time, state, LEFT(info,80) AS info
FROM information_schema.processlist
WHERE command <> 'Sleep' OR time > 60
ORDER BY time DESC;
```

**3. Прицельно по блокировкам (MySQL 8.0) — точный виновник:**
```sql
SELECT * FROM sys.innodb_lock_waits\G
```
или:
```sql
SELECT
  r.trx_mysql_thread_id AS waiting_thread,
  b.trx_mysql_thread_id AS blocking_thread,
  TIMESTAMPDIFF(SECOND, b.trx_started, NOW()) AS blocker_age_sec,
  b.trx_query AS blocking_query
FROM performance_schema.data_lock_waits w
JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_engine_transaction_id
JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_engine_transaction_id;
```

**4. Открытые транзакции по возрасту:**
```sql
SELECT trx_id, trx_state,
       TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS age_sec,
       trx_mysql_thread_id AS thread_id, trx_rows_locked, trx_query
FROM information_schema.innodb_trx
ORDER BY trx_started;
```

### Решение
Убить блокирующий коннект:
```sql
KILL <thread_id>;   -- Id из SHOW PROCESSLIST / trx_mysql_thread_id
```
После этого локи спадают сразу, бэкенд отвисает (перезапускать не обязательно).

### Нюансы прав
- `innodb_trx`, `performance_schema.data_lock_waits`, `sys.innodb_lock_waits` требуют привилегии **`PROCESS`**. Без неё — `ERROR 1227 Access denied`.
- Без `PROCESS` юзер видит в `SHOW PROCESSLIST` **только свои коннекты** (того же аккаунта) и может **убивать только потоки своего аккаунта** (без `SUPER`). Если приложение ходит тем же юзером — этого хватает: видишь и пул бэкенда, и свой дебаг-коннект, и можешь его убить.
- Управляемая БД хостинга (Timeweb и т.п.) — OS-root и `sudo mysql` не помогут, локального сокета нет, доступ только через сетевого юзера.
- Различать коннекты по `Host`: твой домашний/рабочий IP = дебаг (убивать можно), внутренний IP сервера/докера = живой бэкенд (не трогать).

### Профилактика
- Не дебажить по живой preprod/общей БД с долгими остановками на брейкпоинтах — использовать локальную базу.
- Брейкпоинты ставить до старта транзакции или после её завершения, не в середине `@Transactional`.
- Попросить выдать `dev_user` привилегию `PROCESS` (read-only, безопасна) — тогда вся диагностика в одну команду.

### Чеклист «сайт не грузится»
1. `docker logs <backend>` → есть `1205 / Lock wait timeout`? → проблема с локами БД.
2. `SHOW PROCESSLIST;` → ищи `Sleep`/`Query` с большим `Time`.
3. (если есть права) `SELECT * FROM sys.innodb_lock_waits;` → точный виновник.
4. `KILL <id>` виновника.

## Это общий способ для всех БД или только MySQL?

**Подход универсален, конкретные команды — нет.** Идея одинакова для всех реляционных БД:
1. найти долго живущую / блокирующую транзакцию,
2. посмотреть граф «кто кого блокирует»,
3. убить виновника.

Но синтаксис, имена системных таблиц и коды ошибок у каждой СУБД свои:

| Действие | MySQL/InnoDB | PostgreSQL | Oracle | SQL Server |
|---|---|---|---|---|
| Список сессий | `SHOW PROCESSLIST` / `information_schema.processlist` | `pg_stat_activity` | `v$session` | `sys.dm_exec_sessions`, `sys.dm_exec_requests` |
| Блокировки | `sys.innodb_lock_waits`, `performance_schema.data_lock_waits` | `pg_locks` + `pg_blocking_pids()` | `v$lock`, `dba_blockers/dba_waiters` | `sys.dm_tran_locks`, `sys.dm_os_waiting_tasks` |
| Открытые транзакции | `information_schema.innodb_trx` | `pg_stat_activity` (`state='idle in transaction'`, `xact_start`) | `v$transaction` | `sys.dm_tran_active_transactions` |
| Убить сессию | `KILL <id>` | `pg_terminate_backend(pid)` / `pg_cancel_backend(pid)` | `ALTER SYSTEM KILL SESSION '<sid>,<serial#>'` | `KILL <spid>` |
| Аналог «idle-дебага» | `Sleep` + большой `Time` | `idle in transaction` + старый `xact_start` | `INACTIVE` сессия с открытой транзакцией | `sleeping` + открытый tran |
| Код ошибки лок-таймаута | `1205` | нет жёсткого (есть `lock_timeout`, `statement_timeout`); дедлок = `40P01` | `ORA-00060` (deadlock), `ORA-30006` | `1222` (lock timeout), `1205` (deadlock) |

**Пример для PostgreSQL** (частый случай — `idle in transaction` после дебага):
```sql
-- кто кого блокирует
SELECT blocked.pid AS blocked_pid, blocking.pid AS blocking_pid,
       blocking.query AS blocking_query,
       now() - blocking.xact_start AS blocker_age
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking
  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid));

-- найти зависшие транзакции
SELECT pid, state, now() - xact_start AS age, query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY xact_start;

-- убить
SELECT pg_terminate_backend(<pid>);
```

**Вывод: методология одна для всех БД, шпаргалку с командами держать под каждую СУБД отдельно.**
