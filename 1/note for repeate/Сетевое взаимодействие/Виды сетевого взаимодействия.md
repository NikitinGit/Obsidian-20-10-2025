# HTTP-коннект и keep-alive

>[!question]- что такое «коннект» физически
> Это **TCP-соединение** — открытый сокет между двумя сторонами, однозначно определяемый **4-кортежем**: `(IP клиента : порт клиента) ↔ (IP сервера : порт сервера)`. Пока сокет открыт с обеих сторон — соединение «живо». HTTP — это то, что *передаётся внутри* этого сокета, а не сам сокет.

>[!question]- как открывается коннект (TCP 3-way handshake, до всякого HTTP)
> Прежде чем уйдёт хоть один HTTP-запрос — рукопожатие TCP:
> ```
> клиент → SYN →      сервер
> клиент ← SYN-ACK ←  сервер
> клиент → ACK →      сервер
> # TCP установлено, можно слать HTTP
> ```
> Для HTTPS дальше ещё TLS-рукопожатие поверх этого.

>[!question]- жизненный цикл коннекта: HTTP/1.0 vs HTTP/1.1 vs HTTP/2
> **HTTP/1.0** — «открыл → запрос → ответ → закрыл», один коннект = один запрос:
> ```
> [TCP open] → GET /a → 200 OK → [TCP close]
> [TCP open] → GET /b → 200 OK → [TCP close]
> ```
> **HTTP/1.1** — **keep-alive по умолчанию**: коннект открывается один раз и переиспользуется:
> ```
> [TCP open] → GET /a → 200 → GET /b → 200 → GET /c → 200 → [TCP close по условию]
> ```
> Заголовки: `Connection: keep-alive`, `Keep-Alive: timeout=75, max=100`. Модель всё равно строго последовательная `запрос→ответ` (сервер сам заговорить не может — см. [[WebSocketStomp]]).
> 
> **HTTP/2** — одно TCP-соединение, внутри много **мультиплексированных** стримов одновременно (убирает head-of-line blocking, который в 1.1 оставался: запросы по одному коннекту шли по очереди).

>[!question]- когда HTTP/1.1-коннект закрывается
> Не «после ответа», а по одному из условий:
> 1. **Idle-таймаут** — нет новых запросов N секунд → сервер закрывает. В nginx это `keepalive_timeout` (дефолт ~75с) — см. [[NGINX]].
> 2. **`Connection: close`** — сторона говорит «это последний запрос».
> 3. **Лимит запросов** на соединение (`max=...`) исчерпан.
> 4. Явное закрытие приложением, ошибка, обрыв сети.
> 
> Физически закрытие — TCP **4-way (FIN)**:
> ```
> A → FIN → B ;  A ← ACK ← B ;  A ← FIN ← B ;  A → ACK → B   # сокет закрыт
> ```
> `RST` — резкий обрыв без этой процедуры (тот самый WS `code 1006`, когда прослойка рубит соединение).

>[!question]- зачем в HTTP/1.1 сделали keep-alive
> Чтобы **не платить заново за установку соединения** на каждом запросе. Что дорого при новом коннекте:
> 1. **TCP 3-way handshake** — минимум **1 RTT** до первого байта данных.
> 2. **TLS-рукопожатие** (HTTPS) — ещё **1–2 RTT** (сертификаты, ключи). Самое дорогое.
> 3. **TCP slow start** — TCP в начале шлёт медленно и «разгоняется»; новое соединение всегда стартует на малой скорости.
> 
> При RTT ~50мс новый HTTPS-коннект = ~150мс только на рукопожатия, ДО первого байта. Страница тянет десятки ресурсов → открывать под каждый новый коннект = катастрофа.
> 
> **Что даёт keep-alive** (переиспользуя уже открытый и разогнанный сокет):
> - нет повторного TCP-handshake (−1 RTT);
> - нет повторного TLS-handshake (−1–2 RTT, самое ценное для HTTPS);
> - TCP уже разогнан (slow start пройден) → полная скорость сразу;
> - меньше нагрузка на сервер: открытие/закрытие сокетов и TLS стоит CPU/памяти, плюс закрытые сокеты висят в `TIME_WAIT` ~1–2 мин, занимая порты.
> 
> **Итог:** keep-alive — оптимизация «переиспользуй дорого установленный канал» (главная экономия — TLS-рукопожатие), а НЕ изменение модели «запрос→ответ». Server push это не даёт — для него нужен [[WebSocketStomp|WebSocket]].
