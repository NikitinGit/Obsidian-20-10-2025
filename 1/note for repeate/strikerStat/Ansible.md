# Ansible в StrikerStat

Ansible — инструмент **автоматизации настройки серверов и выкатки** (configuration management + deployment + orchestration). Не система сборки: он не компилирует код, а описывает *желаемое состояние* серверов и приводит их к нему по SSH. В StrikerStat плейбуки заодно вызывают сборку приложений (`npm build`, клон репо, запуск контейнеров), но суть Ansible — оркестрация: пакеты → код → рендер конфигов → перезапуск сервисов.

Репозиторий плейбуков: `ansible-playbooks` (локально `/home/igor/IdeaProjects/springdoc/ansible-playbooks`). Запуск — через веб-UI **Semaphore** (`ansible.strikerstat.com`). Связка с nginx-конфигом — см. [[WebSocketStomp]] (колаут про то, что серверный nginx генерится из шаблона).

## Компоненты

>[!question]- что такое Ansible (и чем НЕ является)
> Инструмент **автоматизации операций на удалённых серверах**: настройка ОС, установка пакетов, деплой приложений, перезапуск сервисов — по единому сценарию, повторяемо.
> - **agentless** — на целевых серверах ничего ставить не надо, Ansible ходит по обычному **SSH** с управляющего сервера;
> - **декларативный + идемпотентный** — описываешь «как должно быть», а не «пошаговый скрипт»; повторный запуск меняет только то, что разошлось;
> - **НЕ сборщик**: Maven/npm он не заменяет. Но плейбук может *вызвать* сборку как один из шагов.
> Правильная формулировка: не «Ansible собирает проект», а «Ansible настраивает серверы и раскатывает то, что собрано».

>[!question]- Playbook (плейбук) — сценарий автоматизации
> YAML-файл со списком **задач**, выполняемых на выбранных хостах. Это и есть «один сценарий сборки/выкатки».
> Примеры в репо:
> - `deploy-all.yml` — весь стенд (front + php + java + все nginx-конфиги);
> - `deploy-front.yml` — только фронт (+ его nginx);
> - `deploy-java.yml`, `deploy-php.yml`, `deploy-content-admin.yml` — по компонентам;
> - разовые: `setup-app-server.yml`, `letsencrypt.yml`, `ufw.yml`, `mysql.yml`.
> Шапка плейбука задаёт `hosts:` (на ком гонять), `become: yes` (от root), `vars_files:`/`vars:` (переменные).

>[!question]- Task (задача) + Module (модуль) — один шаг
> **Задача** — один шаг плейбука; она вызывает **модуль** — готовую идемпотентную операцию.
> Частые модули (из `deploy-all.yml`):
> - `apt:` — установить пакеты (`nginx`, `nodejs`);
> - `git:` — склонировать/обновить репозиторий приложения;
> - `template:` — отрендерить Jinja-шаблон в файл на сервере;
> - `file:` — создать симлинк/директорию;
> - `command:` / `service:` — выполнить команду / управлять сервисом.
> Модуль сам решает, надо ли что-то менять (idempotent): если состояние уже нужное — статус `ok`, если изменил — `changed`.

>[!question]- Role (роль) — переиспользуемый блок задач
> Папка со стандартной структурой (`tasks/`, `handlers/`, `templates/`, `defaults/`), которую можно подключать из разных плейбуков с параметрами.
> Пример: **`roles/nginx-config`** — универсально раскатывает любой nginx-сайт:
> ```yaml
> # tasks/main.yml
> - template: { src: "{{ nginx_template }}", dest: "/etc/nginx/sites-available/{{ nginx_site }}.conf" }
>   notify: [Проверка конфигурации nginx, Перезагрузка nginx]
> - file: { src: .../sites-available/..., dest: .../sites-enabled/..., state: link }
> ```
> Вызывается из `deploy-all.yml`/`deploy-front.yml` через `include_role` с параметрами `nginx_site: front`, `nginx_template: front-nginx.conf.j2` (и так же для backend-php, backend-java).

>[!question]- Template (шаблон, Jinja2 `.j2`) — конфиг с переменными
> Файл с плейсхолдерами `{{ var }}` и логикой `{% if %}` / `{% for %}`, который модуль `template:` рендерит в конечный файл, подставляя переменные окружения.
> Пример: **`front-nginx.conf.j2`** → `/etc/nginx/sites-available/front.conf`. В нём `server_name {{ domain_name }}`, блок авторизации под `{% if server_env != 'prod' %}`, `allow` из цикла `{% for ip in testing_allowed_ips %}`.
> **Отсюда ключевое правило:** конфиг на сервере — это *результат рендера*; править надо **шаблон в репе**, а не файл на сервере (иначе следующий прогон Ansible затрёт). Подробнее — [[WebSocketStomp]].

>[!question]- Handler (хендлер) — реакция на изменение
> Особая задача, которая запускается **только если её кто-то `notify`-нул**, и только когда связанная задача реально что-то изменила (не на каждый прогон).
> Пример: `roles/nginx-config/handlers/main.yml`:
> - `Проверка конфигурации nginx` → `nginx -t` (register результата);
> - `Перезагрузка nginx` → `service reloaded`, **только если** `nginx -t` прошёл (`when: nginx_test.rc == 0`).
> Смысл: перезагружать nginx не всегда, а лишь когда конфиг сменился и валиден.

>[!question]- Inventory (инвентарь) — где и на чём запускать
> Список **хостов**, сгруппированных по **окружениям** (`staging` / `testing` / `preprod` / `prod`), плюс **переменные для каждого окружения**.
> В StrikerStat inventory **приватный** (там секреты): лежит на управляющем сервере `/root/ansible/inventory/hosts`, **в git НЕ пушится**. Именно тут задаются `server_env`, `domain_name`, `testing_allowed_ips`, git-креды и т.п.
> Выбор окружения при запуске — флаг `-l <env>` (напр. `-l testing`). В Semaphore это раздел **Инвентарь** (у нас — «StrikerStat»).

>[!question]- Variables (переменные) — параметризация
> Значения, подставляемые в плейбуки/шаблоны. Источники (по возрастанию приоритета, упрощённо):
> - **`common-vars.yml`** — общие для всех (`vars_files`);
> - **inventory group_vars** — по окружению (`server_env`, `domain_name`, `testing_allowed_ips`) — приватно;
> - **`vars:` в плейбуке** — локальные (напр. `front_version: main`, `front_repo_url`);
> - **extra-vars / Semaphore Environment** («Группы переменных») — переопределения при запуске.
> Один и тот же шаблон под разные окружения даёт разный результат — за счёт разных переменных в inventory.

>[!question]- Control node + agentless (SSH) — откуда всё запускается
> Ansible установлен на **одном управляющем сервере** (control node). К целевым серверам ходит по **SSH-ключу** без пароля; на самих серверах агент не нужен.
> Плейбуки на control node лежат в `/root/ansible/playbooks/` и **синхронизируются из гит-репы** (тем самым git-токеном, что чинил синьор). Твой ноут в этом не участвует — локально запустить нельзя (нет приватного inventory и ключей).

>[!question]- Идемпотентность — почему безопасно гонять повторно
> Модули меняют только то, что не в желаемом состоянии. Повторный прогон уже настроенного сервера в основном даёт `ok` (без изменений), а не ломает.
> В выводе видно `ok` / `changed` / `failed`. Поэтому `deploy-all.yml` можно перезапускать — он не «пересоздаёт» всё заново, а доводит до нужного состояния.

>[!question]- Semaphore — веб-запускалка поверх Ansible
> UI (`ansible.strikerstat.com`), который прячет ручной `ansible-playbook`. Сущности:
> - **Шаблоны задач** — привязка «плейбук + инвентарь + окружение» (напр. «Testing Deploy» → `deploy-all.yml`, инвентарь StrikerStat); кнопка ▶ = прогон;
> - вкладки окружений **STAGING / TESTING / PREPROD** (= `-l <env>`);
> - **Инвентарь**, **Хранилище ключей**, **Репозитории** (репо `playbooks` — откуда берутся плейбуки), **Группы переменных** (Environment).
> Требует рабочий **git-токен**, чтобы тянуть репозиторий плейбуков и код приложений.

>[!question]- Как это собирается в проекте (поток выкатки)
> 1. Правишь плейбук/шаблон в репе `ansible-playbooks`, мёржишь в **`main`**.
> 2. В Semaphore жмёшь ▶ на шаблоне нужного окружения (напр. «Testing Deploy» = `deploy-all.yml`, `playbooks@main`).
> 3. Semaphore/control node тянет свежий `main` и гонит плейбук на серверах окружения.
> 4. Плейбук по шагам: `apt` пакеты → `git` клон кода приложения (`version`/`front_version`) → `npm build` → `include_role nginx-config` (рендер `.j2` → `nginx -t` → reload).
> Итог: Ansible **оркестрирует и сборку приложения, и настройку сервера** — но «сборка» это лишь один из шагов сценария, а не суть инструмента.
