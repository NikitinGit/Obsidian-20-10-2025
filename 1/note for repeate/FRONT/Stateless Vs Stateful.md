>[!question]- Что значит «HTTP stateless» — сервер правда ничего не помнит?
>Фраза «HTTP stateless» часто понимается буквально — как будто сервер физически не может ничего запомнить. Это неточно.
>
>**Правильнее:** сам протокол HTTP не содержит встроенного механизма хранения состояния между запросами. Каждый запрос для HTTP — отдельная, независимая транзакция:
>```
>Запрос №1 → сервер обработал → локальные переменные сброшены
>Запрос №2 → сервер начинает с нуля
>```
>HTTP не говорит серверу: «этот запрос пришёл от того же клиента, что предыдущий».
>
>**Но сервер может хранить состояние сам — через логику приложения.** Именно это и делает сессия:
>1. Клиент логинится → сервер создаёт запись в памяти/Redis/БД: `{sessionId: "A8F7", user: "Ivan", role: "admin"}`
>2. Сервер отдаёт клиенту только ключик: `Set-Cookie: JSESSIONID=A8F7`
>3. Следующий запрос клиент шлёт с `Cookie: JSESSIONID=A8F7`
>4. Сервер по ключику находит свою запись → знает, кто это
>
>Сервер помнит — но **не благодаря HTTP**, а потому что сам сохранил данные и клиент прислал ключ для их поиска. Протокол HTTP здесь ни при чём.
>
>**JWT — другая крайность:** состояние хранится не на сервере, а в самом токене у клиента. Сервер только проверяет подпись и читает данные из токена — никакой записи у себя не ищет. (см. [[JWT]])

>[!question]- Два разных «состояния» — соединение vs приложение
>Это ключевое различие, которое часто путают:
>
>| | Состояние соединения | Состояние приложения |
>|---|---|---|
>| Что это | помнит ли протокол, что два запроса от одного клиента | хранит ли сервер данные о пользователе (сессию, роли) |
>| HTTP | ❌ stateless (каждый запрос независим) | ✅ может хранить (сессии, БД) |
>| WebSocket | ✅ stateful (одно постоянное соединение) | ❌ не хранит автоматически (надо писать код) |
>
>WebSocket — stateful на уровне соединения: сервер знает, что все сообщения по данному сокету от одного клиента. Но «пользователь Иван с ролью admin» — это уже состояние приложения, которое разработчик хранит сам (при открытии соединения проверяет JWT/cookie и сохраняет в памяти сервера).

>[!question]- WebSocket — что это и чем отличается от HTTP
>WebSocket начинается с HTTP-запроса (Upgrade), после чего создаётся **постоянное двустороннее TCP-соединение**:
>```
>1. Клиент → Сервер:  GET /ws
>                     Upgrade: websocket
>                     Connection: Upgrade
>
>2. Сервер → Клиент:  101 Switching Protocols
>
>3. [Теперь канал открыт — оба могут слать сообщения в любой момент]
>
>Клиент ============================= Сервер
>             WebSocket (TCP)
>  → "Привет"
>                ← "Привет, Иван!"
>  → "Ещё сообщение"
>```
>
>**Ключевые отличия от HTTP:**
>
>| | HTTP | WebSocket |
>|---|---|---|
>| Инициатор | только клиент | оба (сервер тоже может слать) |
>| Соединение | закрывается после каждого ответа | остаётся открытым |
>| Overhead | заголовки на каждый запрос (~500 байт) | после Upgrade — минимальный фрейм |
>| Применение | REST API, страницы | чат, игры, биржа, уведомления в реальном времени |
>
>Аутентификация при WebSocket делается **один раз при открытии** соединения (через cookie или `?token=...` в URL — заголовок `Authorization` при Upgrade браузер не поддерживает). Дальше сервер запоминает пользователя для этого соединения.

>[!question]- Какие протоколы stateful по природе
>Протокол stateful = сам протокол определяет последовательность шагов и помнит, на каком шаге находится соединение.
>
>**SSH** — после аутентификации все команды выполняются в контексте одной сессии. Не нужно передавать логин при каждой команде.
>
>**FTP** — после `USER` / `PASS` сервер держит «авторизованный контекст», помнит текущую директорию. Нельзя начать с `LIST` без предварительного логина.
>
>**SMTP** — строгая последовательность команд в рамках одного соединения:
>```
>HELO domain.com
>MAIL FROM: <sender@example.com>
>RCPT TO: <receiver@example.com>
>DATA
>...текст письма...
>.
>QUIT
>```
>Начать с `DATA` без предыдущих шагов — ошибка. Сервер помнит, где находится «диалог».
>
>**Почему веб пошёл по пути stateless (HTTP)?** Масштабирование. Если протокол не хранит состояние, любой сервер в кластере может обработать любой запрос без знания истории.

>[!question]- Почему HTTP stateless упрощает масштабирование
>```
>              Load Balancer
>             /      |      \
>      Server A  Server B  Server C
>```
>Запрос №1 (`GET /users`) → Server A
>Запрос №2 (`GET /posts`) → Server C  ✅ — не важно, кто обработает
>Запрос №3 (`GET /profile`) → Server B  ✅
>
>Серверы не знают друг о друге. Балансировщик свободно распределяет нагрузку. Каждый запрос живёт 20–100 мс и завершается — сервер свободен.
>
>Если бы HTTP хранил состояние соединения — пришлось бы либо отправлять все запросы одного клиента на один сервер, либо синхронизировать состояние между серверами. Оба варианта существенно сложнее.
>
>⚠️ **Уточнение:** HTTP/2 и HTTP/3 используют мультиплексирование (много запросов по одному TCP-соединению) — соединение становится постоянным. Но на уровне запросов HTTP всё равно остаётся stateless: каждый запрос независим, просто летит по одному каналу.

>[!question]- Почему WebSocket сложно масштабировать
>Одно WebSocket-соединение всегда привязано к одному серверу — это физическое ограничение TCP:
>```
>Клиент ================= Server B
>           WebSocket
>```
>Сообщение №1 → Server B ✅
>Сообщение №2 → Server C ❌ (соединение не там)
>
>При миллионе клиентов каждый сервер должен держать сотни тысяч открытых сокетов — это память и ресурсы. Этого нет в HTTP, где запрос живёт миллисекунды.
>
>**Три способа масштабировать WebSocket:**
>
>**1. Sticky Sessions** — балансировщик запоминает, что клиент Иван → всегда Server B. Все новые соединения Ивана идут туда же. Самый простой вариант, но создаёт «горячие» серверы.
>
>**2. Брокер сообщений (Redis Pub/Sub, Kafka, RabbitMQ)**
>```
>Иван → Server A
>Пётр → Server B
>Анна → Server C
>          ↕         ↕         ↕
>              Redis Pub/Sub
>```
>Иван написал в чат → Server A публикует в Redis → Redis раздаёт Server B и C → те доставляют Петру и Анне. Серверы не знают друг о друге напрямую, но синхронизируются через брокер.
>
>**3. Внешнее хранилище состояния** — вместо хранения «кто в какой комнате» в памяти Server A, всё пишется в Redis/БД. Тогда любой сервер может прочитать это состояние. WebSocket-соединения всё равно привязаны, но бизнес-логика — нет.

>[!question]- SSE (Server-Sent Events) — когда WebSocket избыточен
>WebSocket двусторонний — клиент и сервер оба шлют сообщения. Но часто нужно только **сервер → клиент** (уведомления, обновления ленты, прогресс задачи).
>
>Для этого есть **SSE** — однонаправленный постоянный HTTP-поток:
>```java
>// Spring — эндпоинт SSE
>@GetMapping(value = "/notifications", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
>public SseEmitter notifications() {
>    SseEmitter emitter = new SseEmitter();
>    // сохранить emitter, слать события при необходимости
>    return emitter;
>}
>```
>```js
>// Браузер — слушать события
>const es = new EventSource('/notifications');
>es.onmessage = e => console.log(e.data);
>```
>
>| | WebSocket | SSE |
>|---|---|---|
>| Направление | двустороннее | только сервер → клиент |
>| Протокол | ws:// / wss:// | обычный HTTP |
>| Переподключение | вручную | браузер сам переподключается |
>| Применение | чат, игры | уведомления, live-feed, прогресс |
>
>SSE проще масштабируется (обычный HTTP, балансировщик работает как с обычными запросами с поправкой на keep-alive), но не заменяет WebSocket там, где нужен диалог.
